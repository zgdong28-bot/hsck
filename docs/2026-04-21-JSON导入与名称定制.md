## 问题

1. CF Worker 源聚合器缺少直接导入 JSON 的方式。目前只能通过 `POST /admin/sources` 逐个添加 URL，无法批量导入多仓 JSON 或粘贴单仓配置内容。
2. 源配置提供方在 site name 中嵌入推广文字（如"关注公众号XXX"），TVBox 左上角和站点列表直接显示这些垃圾文字，无法清除或替换。

## 根因分析

### 功能 1
- 当前 `manual_sources` 存储 `{ name, url }` 数组，系统设计为 URL-based
- `fetcher.ts` 已有多仓检测逻辑（`isMultiRepoConfig`），但只在聚合运行时使用，未暴露为导入入口
- 缺少接受 JSON 内容（非 URL）的 API 端点

### 功能 2
- TVBox 左上角名字来自 `HomeActivity.java:293` 的 `home.getName()`，数据源是配置 JSON `sites` 数组中 home 源的 `name` 字段
- App 代码对 `name` 仅做 `.trim()`，无任何过滤/清洗逻辑
- 推广文字 100% 来自源配置提供方，非 app 硬编码

## 方案

### 功能 1：JSON 直接导入

**新增 API 端点 `POST /admin/sources/import`**

请求体：`{ input: string }`，input 可以是：
- HTTP(S) URL → 后端 fetch 获取内容
- 原始 JSON 字符串 → 直接解析

**处理流程：**
1. 判断 input 是 URL 还是 JSON 内容
   - 以 `http://` 或 `https://` 开头 → URL，fetch 获取内容
   - 否则 → 尝试解析为 JSON
2. 使用 `parseConfigJson` 解析
3. 使用 `isMultiRepoConfig` 检测格式：
   - **多仓**：调用 `extractMultiRepoEntries` 提取所有子 URL，与现有 `manual_sources` 去重后批量添加
   - **单仓且来自 URL**：将原始 URL 添加到 `manual_sources`（去重）
   - **单仓且是粘贴内容**：存储到 KV（key = `inline_config_{timestamp}`），添加 `inline://{key}` 到 `manual_sources`
4. 返回：`{ type: 'multi'|'single', added: number, duplicates: number, sources: string[] }`

**`fetchConfigs` 修改：**
- 检测 `inline://` scheme 的源
- 从 KV 读取对应 key 的 JSON 内容，直接作为 SourcedConfig 返回
- 不走 HTTP fetch

**Admin UI：**
- 新增"导入配置"区域，位于 Sources 区域上方
- 包含一个 textarea（粘贴 JSON 或 URL）+ 导入按钮
- 导入后显示结果：检测到的格式、添加数量、跳过的重复数量

### 功能 2：源名称定制

**新增模块 `cleaner.ts`**

名称变换管道（按顺序执行）：
1. **清洗推广文字**：匹配默认正则模式，执行删除或替换
2. **清理残留分隔符**：去掉首尾的 `|｜/／-—·` 等分隔符，压缩连续分隔符
3. **添加前缀/后缀**：如果配置了，拼接到站点名
4. **空名 Fallback**：如果清洗后名称为空，使用 `site.key` 作为回退

**默认清洗正则（硬编码）：**
```
关注.*?公众号[：:\s]*[a-zA-Z0-9\u4e00-\u9fa5_-]*
公众号[：:\s]*[a-zA-Z0-9\u4e00-\u9fa5_-]+
[Vv][Xx][:：\s]*[a-zA-Z0-9_-]+
加群[：:\s]*\d+
QQ群?[：:\s]*\d+
微信[：:\s]*[a-zA-Z0-9_-]+
[Tt]elegram[：:\s]*@?[a-zA-Z0-9_]+
[Tt][Gg][：:\s]*@?[a-zA-Z0-9_]+
免费.*?观看
更新.*?地址[：:\s]*\S+
最新.*?地址[：:\s]*\S+
备用.*?地址[：:\s]*\S+
```

**配置数据模型（存 KV `name_transform`）：**
```typescript
interface NameTransformConfig {
  prefix?: string;           // 全局前缀，如 "【RioTV】"
  suffix?: string;           // 全局后缀，如 " · 精选"
  promoReplacement?: string; // 推广文字替换为此（空/不设 = 直接删除）
  extraCleanPatterns?: string[]; // 额外清洗正则（叠加到默认规则上）
}
```

**聚合管道注入点：**
- 在 `aggregator.ts` 的 Step 5（`cleanEmptyEntries` + `cleanLocalRefs`）之后调用
- 只处理 `site.name`，不动 parse/live

**Admin UI：**
- 新增"名称定制"区域
- 4 个输入框：前缀、后缀、推广替换文字、额外正则（textarea，每行一条）
- 保存按钮 → `PUT /admin/name-transform`
- 预览功能（可选）：输入一个站点名，实时显示变换结果

**API 端点：**
- `GET /admin/name-transform`：读取当前配置
- `PUT /admin/name-transform`：更新配置

## 风险点

- **inline:// 配置无 base URL**：粘贴的单仓 JSON 中如有相对路径（`./spider.jar`），normalizeConfig 无法解析。可接受——相对路径本身就需要 base URL，用户应粘贴完整配置或提供 URL。
- **正则过度匹配**：如站点名就叫"公众号影视"，可能被误伤。通过精确匹配"关注+公众号"等号召语句模式降低风险，单独的"公众号"三个字不会被匹配。
- **导入大量重复源**：多次导入同一多仓 JSON。解法：添加前按 URL 去重，返回 duplicates 计数。
- **清洗后 name 为空**：全是推广文字的站点名。Fallback 到 `site.key`。

## 验证方式

- 导入一个已知的多仓 JSON URL（storeHouse 格式），验证所有子源 URL 被正确提取和添加
- 导入一个 urls 格式的多仓 JSON，验证兼容
- 粘贴一个单仓配置 JSON 内容，验证 inline 配置在聚合时被正确读取和合并
- 重复导入同一 JSON，验证去重逻辑生效
- 配置前缀"【Test】"，触发聚合，验证所有 site name 都带前缀
- 配置推广替换文字"精选"，验证"关注公众号XXX"被替换为"精选"
- 输入一个全是推广文字的站点名，验证 fallback 到 key

## 讨论记录

- **方案 A（纯前端解析）被否决**：前端 fetch 外部 URL 存在 CORS 问题，且需要重写后端已有的解析逻辑
- **名称清洗范围**：确认只清洗 `site.name`，parse/live name 推广文字罕见
- **左上角品牌名**：确认用全局前缀方案（纯 CF Worker 侧），不需要修改 app 代码
- **可配置规则**：默认硬编码规则 + KV 存储额外规则，不做投机抽象但预留扩展口
