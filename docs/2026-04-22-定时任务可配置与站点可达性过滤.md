## 问题

1. **定时任务不可配置**：聚合频率硬编码在环境变量/wrangler.toml 中，用户无法在 Admin UI 动态调整
2. **聚合后站点 API 不通**：聚合器不验证站点 API 可达性，死站点照样收进输出，TVBox 客户端加载时显示空内容

## 根因分析

### 问题 1
- Node.js：`CRON_SCHEDULE` 环境变量控制，改了需要重启
- CF Worker：`wrangler.toml` 的 `[triggers].crons` 部署时写死，运行时无法修改（平台限制）

### 问题 2
- `aggregator.ts` Step 6 的站点测速只在 Node.js 本地模式运行，CF 模式完全跳过
- 测速结果只用于追加 `[0.4s]` 标签到站点名称，**不移除不可达站点**
- `speedtest.ts` 的 `appendSpeedToName` 对 `speedMs === null` 的站点不做任何处理

## 方案

### 功能 1：定时任务 Admin UI 可配置

**存储**：新增 KV key `cron_interval`，存储间隔分钟数

**下拉选项**：
| 显示文本 | 值（分钟） | cron 表达式（Node.js） |
|---------|-----------|----------------------|
| 每 1 小时 | 60 | `0 */1 * * *` |
| 每 3 小时 | 180 | `0 */3 * * *` |
| 每 6 小时 | 360 | `0 */6 * * *` |
| 每 12 小时 | 720 | `0 */12 * * *` |
| 每天一次 | 1440 | `0 5 * * *` |

**Node.js 侧**：
- 启动时读取 `cron_interval`，转换为 cron 表达式
- Admin 修改间隔后，通过 `/admin/cron-interval` PUT API 更新 KV
- 主进程提供 `rescheduleCron()` 函数，Admin API 调用后重建 cron job

**CF Worker 侧**：
- `wrangler.toml` cron 改为 `0 * * * *`（每小时触发）
- `scheduled()` handler 读取 `cron_interval` + `last_update`
- 如果 `now - last_update < interval`，跳过本次执行

### 功能 2：不可达站点过滤

**核心改造**：`aggregator.ts` Step 6 从"测速标名字"升级为"测速 + 过滤不可达 + 标名字"

**统一流程（CF + Node.js）**：
1. 对合并后的 `merged.sites` 执行 `batchSiteSpeedTest`
2. 新增 `filterUnreachableSites(sites, speedMap)` 函数：移除 speedMs === null 的站点
3. 安全阀：过滤后站点数 < 原始的 30%，回退到不过滤（防网络抖动误杀）
4. 保留 `appendSpeedToName` 逻辑，对可达站点追加延迟标签

**CF 优化策略**：
- 全量并发（不分批），3 秒超时
- CF scheduled event 允许 1000 子请求，足够测 800+ 站点
- 如果可测站点数超过子请求剩余名额（约 900），截断只测前 800 个

**安全阀逻辑**：
```
过滤后站点数 / 原始站点数 < 0.3 → 回退不过滤，日志警告
```

## 改动范围

### 功能 1（定时任务）
- `src/core/config.ts`：新增 `KV_CRON_INTERVAL` 常量
- `src/core/admin.ts`：设置 tab 增加聚合频率下拉框 UI
- `src/routes.ts`：新增 `GET/PUT /admin/cron-interval` API
- `src/node-entry.ts`：cron 动态重建逻辑，导出 `rescheduleCron()` 接口
- `src/cf-entry.ts`：`scheduled()` 增加间隔检查逻辑
- `wrangler.toml`：cron 改为 `0 * * * *`

### 功能 2（站点过滤）
- `src/aggregator.ts`：Step 6 改造——CF 和 Node.js 统一走测速过滤
- `src/core/speedtest.ts`：新增 `filterUnreachableSites()` 函数

## 风险点

- CF Worker `wrangler.toml` cron 从每天一次改为每小时一次，会增加 Worker 调用次数（但跳过逻辑确保不执行真正的聚合）
- 站点测速依赖网络状态——某些站点可能因 CF 出口 IP 被限制而误判为不可达
- 安全阀阈值 30% 可能需要根据实际情况调整

## 验证方式

### 功能 1
- Admin UI 修改频率后，检查 KV 中 `cron_interval` 值是否更新
- Node.js：修改频率后观察 cron 是否按新频率触发
- CF Worker：手动触发 scheduled event，检查间隔控制逻辑

### 功能 2
- 聚合后检查日志中 `[speedtest] X/Y sites reachable` 输出
- 对比过滤前后站点数量
- TVBox 客户端验证站点列表中是否还有死站点

## 讨论记录

- 方案 B（取消 cron 改 setInterval）被否决：失去精确调度能力
- 方案 B（标记但保留不可达站点）被否决：用户体验改善有限
- 方案 C（分级容忍）被否决：复杂度高，简单过滤 + 安全阀已够用
- CF 站点测速原以为受限于 CPU 时间，实际 scheduled event 子请求限制 1000 足够用，全量并发可行
