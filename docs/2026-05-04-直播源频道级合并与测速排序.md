## 问题

TVBox 非独立直播模式只取 `lives[0]`。聚合器把 N 个配置源的 lives 原样堆到 `lives[]` 数组里，
`lives[0]` 不可达 → 整个直播崩溃。即便保证可达，单源挂掉后用户也无法切换。

## 根因分析

1. `src/core/merger.ts:66` 把所有 `config.lives` 无脑 push 进 `allLives`
2. `src/core/dedup.ts:66 deduplicateLives()` 只按 URL 去重，不测速、不排序
3. `src/aggregator.ts:194` extraLives（admin 手动源，已测速）追加到末尾，到不了 `lives[0]`
4. TVBox 协议第 3 层 `TVBoxLiveChannel.urls[]` 原生支持多 URL fallback，
   但当前输出 `lives[0].url` 指向外部 m3u 文件 —— m3u 下载失败时 fallback 链路直接中断

## TVBox 协议事实依据（来自 TVBox-Pro/ApiConfig.java）

TVBox 存在两种直播配置格式，通过 `lives.contains("type")` 字符串匹配区分：

| 格式 | 触发条件 | 处理方式 |
|------|---------|---------|
| **Native（老格式）** | lives JSON 不含 `"type"` | `loadLives()` 遍历所有 group，原生多源切换 ✅ |
| **FongMi（新格式）** | lives JSON 含 `"type"` | 只取 `get(0)`，单源 ❌ |

Native 格式实际结构：
- `lives` 数组每个元素 **就是 group**，不是包装层
- 每个 group：`{group: string, channels: TVBoxLiveChannel[]}`
- 每个 channel：`{name: string, urls: ["http://xxx$源名", ...]}`
- URL 用 `$` 分隔播放地址和源名，TVBox 播放 UI 显示"源A / 源B"让用户切换

## 方案（D+：Native 格式 + 双层测速）

CF Worker 和 Node/Docker 都走 Native 格式输出，频道级 URL 测速仅在 Node/Docker 启用。

### 主聚合流程（每次跑）

1. 下载所有源的 m3u/txt 内容（复用现有 fetch + edge proxy 回退）
2. 解析为 `{group, name, urls[], source}` 频道列表
3. 频道名规范化：trim 空白、去 `HD/高清/蓝光/4K/1080P` 后缀、繁简转换
4. 按规范化名合并 urls 到全局频道表，同源 URL 去重
5. 查询频道测速缓存（仅 Node/Docker）
6. 对每个频道 `urls[]` 排序：
   - 有缓存 → URL 级延迟排序
   - 无缓存 / CF → 源级 speedMap 粗排序
7. URL 拼接 `$源名` 格式
8. 组装 `TVBoxLiveGroup[]` 输出，`merged.lives` 直接赋值

### 独立后台任务（仅 Node/Docker）

- **独立 cron**：每 12 小时触发，与主聚合解耦
- **独立模块**：`src/core/channel-probe.ts` 完全独立
- **独立存储**：`KV_CHANNEL_SPEED_MAP` / `KV_CHANNEL_PROBE_STATUS`
- 并发 50，单 URL 5 秒超时
- m3u8：GET 首 1KB 验 `#EXTM3U` + 记录 TTFB
- ts 裸流：GET 首 4KB 验 sync byte `0x47`
- 其他：TCP 连通性
- 结果缓存 7 天，过期清理
- 失败不影响主聚合

## 关键实现约束（Native 格式陷阱）

1. **lives 数组元素 = group**，不是包装层 TVBoxLive
2. **URL 必须带 `$源名`** 后缀，否则 UI 切换不友好
3. **严禁输出 `"type"` 字段** —— TVBox 用字符串匹配判断格式，任何位置出现字面量 `"type"` 会误判为 FongMi 格式导致只取首 group
4. group 对象严格只含 `{group, channels}`，channel 对象严格只含 `{name, urls, logo?}`
5. group 名或 channel 名若含"type"字样（极少见），需过滤或替换

## 风险点

1. **频道名规范化误合**：CCTV-4 欧洲 vs 美洲可能被误合。对策：规范化后名字相同但 tvg-id/tvg-name 不同则保留分离
2. **m3u8 多级 playlist**：首 1KB 只拿到主 playlist，测速是近似值。接受
3. **反爬封禁**：12 小时间隔 + 同源 URL 串行化
4. **存储膨胀**：合并频道树 + 测速缓存约 MB 级。Node SQLite/JSON 无压力；CF 不跑此任务
5. **主聚合耗时增加**：下载 ~50 个 m3u + 解析，新增 10-30 秒。接受
6. **FongMi TV 分支兼容性**：未确认是否认 Native 格式。当前用户用 TVBox-Pro 已确认支持，其他分支作为后续话题
7. **`"type"` 字段泄漏**：必须在序列化前做全 JSON 字符串扫描校验，出现则报警

## 验证方式

1. **合并正确性**：
   - 聚合后 `lives` 为 `TVBoxLiveGroup[]`，无包装层
   - 所有 group 对象字段严格只有 `{group, channels}`
   - 同名频道 `urls.length >= 合并前出现次数`
   - 全量 JSON 字符串不包含 `"type"`

2. **URL 格式**：
   - 所有 `urls[]` 元素形如 `http://xxx$源名`
   - `$` 右侧源名非空

3. **排序正确性**：
   - 首次聚合（无缓存）：`urls[]` 顺序与源级 speedMap 一致
   - 频道测速后：`urls[]` 顺序与 URL 级 speedMap 一致

4. **播放 fallback**：
   - TVBox-Pro 播 CCTV-1，首个 URL 挂时自动切下一个（内核原生）
   - 播放 UI 显示多个源名可手动切换
   - 全部挂时才报错

5. **运维面板**：
   - admin 面板显示频道测速状态
   - 手动触发 → 实时进度
   - 关闭开关 → 主聚合降级为粗排序仍能正常跑

6. **性能**：
   - 主聚合总耗时增量 ≤ 30 秒
   - 频道测速任务 3000 URL ≤ 5 分钟

## 讨论记录

- 方案 A（仅 lives 数组测速排序）、方案 B（下载+合并 m3u 文件代理）、方案 C（A+B 混合）被否决
- 方案 D+ 核心洞察：`TVBoxLiveChannel.urls[]` 原生支持多 URL fallback，协议层面就该这么做
- CF Worker 子请求和 CPU 限制排除了 CF 侧跑频道测速 → CF 退化为源级粗排序
- 频道名规范化不做中英映射表（用户决定），只做基础清洗
- 用户要求代码独立、方便运维 → 新模块、独立 cron、独立 admin 面板、独立存储 key
- 扒 `TVBox-Pro/ApiConfig.java:590-821` 确认 Native 格式支持：`loadLives()` 原生多源，lives 数组 = groups 列表，URL 用 `$` 分隔源名
- 发现致命陷阱：`!lives.contains("type")` 字符串匹配检测，输出必须严格过滤 `"type"` 字段
- 参考 `TVBox-Pro/docs/2026-04-28-直播多源切换支持.md` 是 App 侧方案，D+ 是协议正统替代
