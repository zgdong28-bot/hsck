## 问题

聚合源中 117 个源涉及网盘 cookie/token 配置（夸克、阿里、Bilibili、115 等），当前存在两个痛点：

1. **重复配置**：每个源各自在 ext 中配 cookie，用户需要在多处填写相同的凭证，且电视端操作不便
2. **安全风险**：47 个 token.json 派源会获取全部网盘凭证并连接第三方服务器，其中 5-25 个源可能将 cookie 转发给作者服务器

## 根因分析

1. TVBox 没有"全局 cookie 存储"，每个 Spider 只读自己的 ext 参数
2. 不同 Spider 的 ext 格式不统一（字段名、结构各异），无法在客户端侧做统一管理
3. token.json 派将所有网盘凭证打包在一起，Spider 无差别获取全部凭证（即使只需要夸克 cookie）

**但聚合器在输出 JSON 之前拥有完整的改写能力**——可以在服务端按源注入 cookie、按风险等级控制投放。

### 源的风险分级规则

基于本次分析的数据，建立以下自动分级逻辑：

| 等级 | 判定条件 | 数量 | Cookie 注入策略 |
|------|---------|------|---------------|
| **安全** | ext 无 cookie 字段 + 无 site URL（A类） | 62 | 不需要注入（源不接受 cookie） |
| **安全** | ext 有 cookie 字段 + 无第三方 site URL（B类） | 48 | **自动注入** |
| **低风险** | token.json 派 + proxy=noproxy 或 proxy=db | 9 | **自动注入**（明确不走代理） |
| **高风险** | token.json 派 + proxy=proxy | 5 | **默认不注入**，需用户确认 |
| **未审计** | token.json 派 + proxy=null 或其他 | ~20 | **默认不注入**，需用户确认 |
| **低风险** | ext 有网站 session cookie + site URL（C类） | 6 | **自动注入**（非网盘 cookie） |

### 第三步：源风险分级引擎

**文件**：`src/core/credential-risk.ts`（新建）

```typescript
export type RiskLevel = 'safe' | 'low' | 'high' | 'unaudited';

export interface SourceRiskAssessment {
  siteKey: string;
  api: string;
  riskLevel: RiskLevel;
  reason: string;
  neededPlatforms: CloudPlatform[];  // 该源需要哪些网盘的凭证
  thirdPartyDomains: string[];       // 连接的第三方域名
}

/**
 * 分析单个源的风险等级
 *
 * 判定逻辑（零网络请求，纯本地解析 ext 字段）：
 * 1. 解析 ext 结构，提取 cookie 字段名和 site URL
 * 2. ext 无 cookie 相关字段 → safe（A类，源不接受凭证）
 * 3. ext 有 cookie 字段 + 无第三方 site URL → safe（B类，直连官方）
 * 4. token.json 引用 → 解析 $$$ 分隔格式提取 proxy 参数:
 *    ext 格式: token.json_url$$$site_url$$$proxy$$$num$$$config
 *    例: ...token.json$$$https://duopan.fun/$$$noproxy$$$1$$$./json/wogg.json
 *    例: ...token.json$$$https://tudou.lvdoui.top/$$$proxy$$$1$$$./lib/wog.json
 *    - proxy 字段在第 3 段（0-indexed: segments[2]）
 *    - noproxy / db → low
 *    - proxy → high
 *    - null / 其他 → unaudited
 * 5. ext 有 cookie + 有 site URL → low（C类，网站 session）
 */
export function assessSourceRisk(site: TVBoxSite): SourceRiskAssessment
```

同时提供风险白名单/黑名单机制：
```typescript
export interface CredentialPolicyConfig {
  // 用户手动放行的高风险源 key（确认愿意承担风险）
  allowedHighRiskKeys: string[];
  // 用户手动拉黑的源 key（即使安全也不注入）
  deniedKeys: string[];
}
```

**验证**：对当前 882 个源跑分级，确认输出与本文分析一致（62 A + 48 B + 9 low + 5 high + ~20 unaudited + 6 C）。

### 第四步：Cookie 注入引擎

**文件**：`src/core/credential-injector.ts`（新建）

核心逻辑：根据源的 Spider 类型和 ext 结构，将凭证注入到正确的 ext 字段。

**关键原则：保持 ext 原有格式**
- 输入是 string → 输出必须是 string（Spider 端按 string 解析）
- 输入是 object → 输出必须是 object
- 注入时先探测格式（string → 尝试 JSON.parse → 按分隔符解析），注入后恢复原格式

```typescript
/**
 * Spider 类型 → ext 注入规则映射表
 *
 * 每个规则定义：
 * - 哪些 CloudPlatform 的凭证需要注入
 * - 注入到 ext 的哪个字段
 * - ext 的格式（JSON object / token.json URL / $$$ 分隔字符串）
 */
export interface InjectionRule {
  apiPattern: string | RegExp;         // 匹配 site.api
  platforms: CloudPlatform[];          // 需要的网盘平台
  inject: (ext: any, credentials: Map<CloudPlatform, CloudCredential>) => any;
}
```

内置规则（覆盖分析出的主要 Spider 类型）：

```typescript
const BUILTIN_RULES: InjectionRule[] = [
  // csp_Bili: ext.cookie = bilibili cookie
  {
    apiPattern: /^csp_Bili/,
    platforms: ['bilibili'],
    inject: (ext, creds) => {
      const obj = parseExt(ext);
      obj.cookie = creds.get('bilibili')?.credential.cookie || '';
      return obj;
    },
  },

  // csp_Wobg (token.json 派): 替换 token.json URL 为聚合器自托管版本
  {
    apiPattern: 'csp_Wobg',
    platforms: ['aliyun', 'quark', 'uc', 'pan115', 'thunder', 'pikpak'],
    inject: (ext, creds) => {
      // 将 ext 中的 token.json URL 替换为聚合器的 /credential/token.json
      return ext.replace(/https?:\/\/[^\$]+token\.json/, '{baseUrl}/credential/token.json');
    },
  },

  // csp_Mogg (含 cookie 版): ext.cookie = quark, ext.token = aliyun, ...
  {
    apiPattern: 'csp_Mogg',
    platforms: ['quark', 'aliyun', 'uc', 'tianyi', 'baidu', 'pan123', 'thunder'],
    inject: (ext, creds) => {
      const obj = parseExt(ext);
      if ('cookie' in obj) obj.cookie = creds.get('quark')?.credential.cookie || '';
      if ('token' in obj) obj.token = creds.get('aliyun')?.credential.refresh_token || '';
      if ('uccookie' in obj) obj.uccookie = creds.get('uc')?.credential.cookie || '';
      if ('tyitoken' in obj) obj.tyitoken = creds.get('tianyi')?.credential.cookie || '';
      if ('dutoken' in obj) obj.dutoken = creds.get('baidu')?.credential.cookie || '';
      if ('p123token' in obj) obj.p123token = creds.get('pan123')?.credential.token || '';
      if ('tuctoken' in obj) obj.tuctoken = creds.get('thunder')?.credential.token || '';
      return obj;
    },
  },

  // ... 其他 Spider 的注入规则
];
```

聚合器自托管 token.json 端点：
```
GET /credential/token.json
```
返回格式与公共 token.json 一致，但只填充用户已登录的网盘凭证。对于高风险源，可进一步按源的 `neededPlatforms` 裁剪——只给该源需要的凭证，不给多余的。

**验证**：注入前后 diff ext 字段，确认只有 cookie 相关字段被修改，其他字段不变。

### 第五步：集成到聚合流水线

**文件**：`src/aggregator.ts`

在 Step 5.5（名称定制）之后、Step 6（站点测速）之前插入：

Step 5.7: 网盘凭证注入（NEW）

```typescript
// Step 5.7: 网盘凭证注入
const credentialConfig = await loadCredentialPolicy(storage);
const credentials = await loadCredentials(storage);

if (credentials.size > 0 && merged.sites) {
  console.log('[aggregation] Step 5.7: Injecting cloud credentials...');

  const report = { injected: 0, skippedSafe: 0, skippedHighRisk: 0, skippedUnaudited: 0 };

  for (let i = 0; i < merged.sites.length; i++) {
    const site = merged.sites[i];
    const risk = assessSourceRisk(site);

    if (risk.riskLevel === 'safe' && risk.neededPlatforms.length === 0) {
      report.skippedSafe++;  // A类：源不需要凭证
      continue;
    }

    if (risk.riskLevel === 'high' || risk.riskLevel === 'unaudited') {
      if (!credentialConfig.allowedHighRiskKeys.includes(site.key)) {
        risk.riskLevel === 'high' ? report.skippedHighRisk++ : report.skippedUnaudited++;
        continue;  // 高风险/未审计源，用户未放行，不注入
      }
    }

    // 执行注入
    const rule = findMatchingRule(site);
    if (rule) {
      merged.sites[i] = { ...site, ext: rule.inject(site.ext, credentials) };
      report.injected++;
    }
  }

  console.log(
    `[aggregation] Credentials: ${report.injected} injected, ` +
    `${report.skippedSafe} not needed, ` +
    `${report.skippedHighRisk} high-risk blocked, ` +
    `${report.skippedUnaudited} unaudited blocked`
  );
}
```

**验证**：跑完整聚合，检查输出 JSON 中安全源的 ext 已填入 cookie，高风险源的 ext 保持原样。

## 风险点

1. **扫码 API 变更**：网盘厂商可能随时更改登录 API 端点或参数，需要维护更新。缓解：参考 AList 社区的实现，跟进其 API 更新
2. **Token 过期**：阿里云盘 refresh_token 有有效期，需要定时刷新。缓解：聚合器 cron 任务中检查凭证有效性，过期提醒用户重新扫码
3. **注入规则不完整**：新增 Spider 类型可能不在规则表中。缓解：未匹配规则的源不注入，不会出错；admin UI 显示"未覆盖"状态，用户可手动添加规则
4. **凭证安全**：凭证存储在本地 SQLite/KV 中，需要加密。缓解：使用 AES-GCM 加密，独立 256-bit 随机密钥存 KV（`credential_encryption_key`），与 ADMIN_TOKEN 解耦，改密码不影响凭证
5. **proxy=null 的分级准确性**：~20 个源的默认行为未知，保守处理为"未审计"。缓解：后续可通过反编译 JAR 或抓包确认，逐个调整等级

## 讨论记录

- **为什么不在 TVBox 侧做全局 cookie？** TVBox 的 Spider 架构决定了每个 Spider 只读自己的 ext，没有全局共享机制。改 TVBox 需要修改 Spider 接口，影响所有 jar 的兼容性。聚合器侧做注入是零侵入方案
- **为什么高风险源默认不注入而非完全禁止？** 用户可能知道某个源的作者可信，或者愿意用专用账号承担风险。禁止不如告知+选择
- **为什么自托管 token.json 而非直接改写 ext？** csp_Wobg 的 ext 是 `$$$` 分隔的字符串格式，其中第一段就是 token.json URL。替换为聚合器自己的 URL 是最干净的方式，不需要理解整个 ext 的复杂格式
- **凭证按需裁剪的意义**：即使是安全源（如 noproxy 模式），也只给它需要的凭证。csp_Bili 只拿 Bilibili cookie，不会拿到夸克/阿里的。相比原始 token.json 的全量打包，显著缩小泄露面
- **为什么加密密钥独立于 ADMIN_TOKEN？** ADMIN_TOKEN 是鉴权凭证，用户可能因安全原因更换。如果加密密钥派生自 ADMIN_TOKEN，改密码就会导致所有已存凭证不可解密，用户需要重新扫码全部平台。独立密钥（方案 A）彻底解耦鉴权和加密两个关注点
- **为什么 token.json 用 `/credential/token.json` 而非 `/api/token.json`？** `/api/:key` 是 MacCMS 边缘代理的通配路由，token.json 会被截获。换路径彻底避免路由优先级问题，不依赖注册顺序
- **proxy 参数确认在 ext 字符串中**：csp_Wobg 的 ext 是 `$$$` 分隔格式，proxy 在第 3 段。风险分级引擎零网络请求即可完成分级，不需要 fetch token.json
