## 问题

Review 发现方案 D+ 的两个高危问题 + 两个中等问题：

1. 🔴 URL query string 里的 `type=` 在 Native 格式下泄漏到最终 JSON，触发 TVBox
   `lives.contains("type")` 子串匹配，误判为 FongMi 格式 → 只取 lives[0] →
   方案 D+ 彻底失效
2. 🔴 Step 6.5 在 CF Worker 下会 fetch N 个 m3u 文件，叠加 aggregator 主流程
   已消耗的子请求，CF 免费版 50 子请求上限会爆，整个聚合崩溃
3. 🟡 CF 下无 probe 消费者仍写 `KV_CHANNEL_MERGED_TREE`，浪费 CF KV 每天
   1000 写额度
4. 🟡 `runWithConcurrency` catch 分支 `err as R` 类型不安全（probeSingle 内
   部已全 try/catch，此路径理论不可达，但写法不自洽）

## 根因分析

1. **URL type 泄漏**：`live-merger.ts:306` URL 组装未处理 URL 内的 `type` 子
   串；`L332` 兜底扫描只检查 `"type":` 字段名，漏掉 value 内 `type` 子串
2. **CF 子请求**：`aggregator.ts:296-358` 对 CF 和 Node 走同一条 Step 6.5
   路径，没有降级分支
3. **KV 浪费**：`aggregator.ts:351` 无条件写 `KV_CHANNEL_MERGED_TREE`
4. **runWithConcurrency**：`channel-probe.ts:170` `err as R` 让后续
   `r.kind`/`r.url` 访问有理论 NPE 风险

## 方案

### 1. URL type 泄漏修复（双保险）

**第一道防线** — `live-merger.ts` URL 组装时预防性 encode：
- 新增辅助函数 `scrubUrlType(url)`：把 URL 字符串里所有 `type` 子串（大小写
  不敏感）替换为 `%74ype`（URL-encode `t` 为 `%74`）
- HTTP RFC 3986 规定服务器必须 URL-decode，`%74ype` 请求到服务端等效 `type`
- 在 L306 `urlStrs` 组装时对 `e.url` 先 scrub 再拼 `$源名`

**第二道防线** — `mergeLivesToNative` 末尾 JSON 兜底扫描：
- 扩大 L331 检查：`/"type"\s*:/i`（字段名）仍保留；**新增**对 value 的
  扫描——如果序列化 JSON 仍含 `type` 子串（不带冒号），说明有遗漏
- 若检测到泄漏：对整个 JSON 字符串里所有非字段名位置的 `type` 做
  `%74ype` 替换（用 regex 排除 `"type":` 匹配），再 `JSON.parse` 回对象
- 日志 warn 打印遗漏来源方便排查

### 2. CF 子请求超限修复（CF 完全跳过 Step 6.5）

- `aggregator.ts` Step 6.5 最外层加分支：`if (config.workerBaseUrl) { skip }`
  - CF 部署下 Step 6.5 整段跳过，lives 保持 merger 合并后的 FongMi 格式
  - `lives[0]` 随缘（保留 TVBox 客户端 lives[0] 崩溃风险）
  - 文档注明："CF 免费版子请求上限制约，频道级合并仅在 Node/Docker 部署启用"
- Node/Docker 下 Step 6.5 完整跑，落地方案 D+ 的 Native 格式频道合并

### 3. KV 写入优化（CF 跳过）

- 方案 2 自然解决：CF 既然整个 Step 6.5 跳过，也就不会走到 L351 写入
- 不需要额外改动

### 4. runWithConcurrency 异常兜底（显式 fail）

- `channel-probe.ts:168-172` catch 分支改为：
  ```ts
  results[i] = { url: items[i] as unknown as string, speedMs: 0, kind: 'fail' } as R;
  ```
- 即便理论上不可达，也避免污染统计和缓存

## 改动范围

- `src/core/live-merger.ts`：
  - L305-306 URL 组装前调用 `scrubUrlType(e.url)`
  - 新增 `scrubUrlType(url: string): string` 函数
  - L330-345 兜底扫描扩展：检测 value 内 `type` 子串并 post-process 替换
- `src/core/channel-probe.ts:168-172`：catch 分支改显式 fail 对象
- `src/aggregator.ts:293-358`：最外层加 `if (config.workerBaseUrl) { skip }`
  分支，console.log 明确说明 CF 下跳过原因

## 风险点

1. **URL-encode `type` → `%74ype` 兼容性**：
   - 主流 HTTP 服务器（Nginx / CDN / 负载均衡）都符合 RFC 3986 做 URL-decode
   - 极少数不规范服务器（自研网关）可能 case-sensitive 对比 encoded 字符
   - 缓解：双保险的第一道就生效，失败时兜底也不会比原来更糟
2. **CF 用户体验倒退**：CF 部署保留 lives[0] 崩溃风险；但 CF 部署本来就有
   免费版资源限制，继续用 CF 的用户属于"次优但能用"场景
   - 缓解：admin 面板频道测速 section 在 CF 下已经通过 404 隐藏，用户不会
     看到无效功能
3. **URL 中非 query 位置的 type**：path 里含 `type`（如 `/api/streamtype`）
   也会被 encode，不影响功能但 URL 变长
4. **兜底 scan 性能**：JSON 字符串几 MB，全文 regex 替换 O(N)，可接受
5. **Post-process 破坏 JSON 结构**：替换只作用于字符串 value 内容（排除字段
   名位置），不会破坏 JSON 结构；再 `JSON.parse` 验证通过即安全

## 验证方式

1. **URL type encode**：构造含 `?type=m3u8` 的测试 URL，验证输出 JSON 里
   `urls[]` 元素为 `?%74ype=m3u8$源名`
2. **兜底 scan**：人工往 groups 里塞一条含 `type` 的 name（绕过 scrubTypeLiteral
   的 scope），验证兜底扫描能捕获并 post-process
3. **CF 分支**：本地 `npm run dev:cf` 启动，触发聚合，验证 Step 6.5 日志
   输出 "Skipped on CF" 且 merged.lives 仍为 FongMi 格式
4. **Node 分支**：`npm start` 启动，触发聚合，验证 Step 6.5 完整跑且
   merged.lives 为 Native groups
5. **TVBox 客户端兼容性**：配置聚合器地址到 TVBox，验证 JSON 不被误判
   为 FongMi（展开多个 groups/channels，多 URL 切换可用）
6. **类型检查**：`npx tsc --noEmit` 通过
7. **runWithConcurrency**：mock fn 抛异常，验证 results 元素为 fail 对象
   而非 Error 实例

## 讨论记录

- URL type 修复三选一：双保险 vs 只 encode vs 只 scan
  - 用户选双保险：防御性最强
- CF 降级三选一：完全跳过 vs 硬截断 vs 强撑
  - 用户选完全跳过：CF 保留 lives[0] 风险但不炸；方案 D+ 定位为 Node/Docker
    专享能力
- 原计划让 CF 做源级 speedMs 排序，但审计后发现 aggregator 无 live 源级测
  速（被之前删除的 processLiveSources 带走），新增 live 测速会再次引入子请
  求问题 → 放弃排序，直接降级到"FongMi 原序"
- URL-encode 选 `t → %74` 而非 `p → %70`：两者效果等价，`%74`（t）更直观
