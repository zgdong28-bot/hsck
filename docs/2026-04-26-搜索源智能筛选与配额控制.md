## 问题

搜索配额系统过于复杂：分层优先级 + JAR Registry orphan 检测 + 搜索级去重 + 保底逻辑 + 长尾排序——层层打补丁，仍然误杀主力搜索源。

搜索级去重的核心假设"同 api class = 同搜索结果"不成立——同一个 `csp_PanWebShare` 配不同 ext 站点 URL，搜索结果完全不同。在这个假设上建的去重逻辑越精细越容易误杀。

## 根因分析

1. **搜索级去重不可靠**：同 api class + 不同 ext = 不同搜索后端，去重会误杀。
2. **配额分层是给错误架构打补丁**：复杂优先级解决的是"去重后谁入选"的问题，但去重本身就不该做。
3. **JAR Registry 的 orphanedKeys 混淆概念**："JAR 分析失败" ≠ "不适合搜索"。

## 方案

### 一、搜索管道简化——不做搜索级去重

**核心决策**：放弃搜索级去重，只保留已有的站点级去重（key|api）。搜索配额纯靠 maxSearchable + 用户排序控制。

**新管道：**
```
站点级去重后: ~500 个站点
  → 排除 JS/URL 源搜索（type=3 + api 是 URL → searchable=0）
  → 置顶源排到 sites 数组最前
  → 如设了 maxSearchable，按数组顺序截断
```

**砍掉：**
- `deduplicateSearchSources()` — 整个搜索级去重函数
- `applySearchQuota()` 的分层优先级逻辑
- 保底逻辑、长尾排序、MacCMS 免配额

**保留：**
- `loadSearchQuota()` / `saveSearchQuota()`（KV 配置读写）
- pinnedKeys（置顶源排最前）
- maxSearchable（可选硬上限，默认 0 = 不限）

**新增：**
- JS 源排除：`type=3 && api.startsWith('http')` → `searchable=0`
- 置顶排序：pinned 源在 sites 数组中移到最前（TVBox 先搜这些源，结果秒出）

### 二、砍 JAR Registry

**删除文件：**
- `src/core/jar-registry.ts`
- `src/core/dex-parser.ts`

**恢复投票制 spider 分配：** merger.ts 回到 `selectGlobalSpider()`，不再依赖 DEX 精确类匹配。

**不影响 JAR Proxy：** jar-proxy.ts 的 URL 改写 + 二进制代理完全独立。

### 三、简化后的 `search-quota.ts`

```typescript
applySearchQuota(sites, config, siteSourceMap):
  1. JS 源排除：type=3 + api 是 URL → searchable=0
  2. 置顶排序：pinnedKeys 中的源移到 sites 数组最前
  3. 可选截断：maxSearchable > 0 时，超出的设 searchable=0
  4. 来源标识：searchable=1 的源 name 追加「来源名」
```

### 四、Admin UI 配置

保留搜索配额面板，简化为：
- maxSearchable 输入（默认 0 = 不限）
- 搜索页签：置顶源管理（pin/unpin）
- 简化报告：去重前 X → JS 排除 Y → 截断 Z = 最终 searchable 数

删除：
- JAR Registry 整个面板
- 复杂的搜索配额报告（active/excluded 双表格、dedup 详情）

## 风险点

1. **投票制精度下降**：部分站点可能分配到非最优 spider。实际影响小（原始行为）。
2. **默认不限搜索数**：~500 源全部 searchable，低端盒子搜索 ~3 分钟。用户可设 maxSearchable。
3. **JS 源排除误伤**：type=3 + HTTP URL 极少数可能是合法 API 而非 JS 引擎。概率极低。

## 验证方式

1. 本地 + NAS 聚合：无 JAR Registry 日志，spider 投票制正常，JS 源被排除
2. pinned 源在 sites 数组前面，maxSearchable=0 不截断
3. TVBox 搜索正常，pinned 源结果先出
