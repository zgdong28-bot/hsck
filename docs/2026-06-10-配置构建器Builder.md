## 问题

当前聚合器是"全量合并 + 黑名单过滤"模式，源越多越不稳定（JSON 不规范、站点不可达导致 TVBox 崩溃）。用户需要一个精细化的配置构建工具：从聚合池中挑选/手动添加条目，深度编辑，导出完全离线的本地包（config.json + JAR）。

## 方案概述

新建 `/builder` 页面（Node.js only），作为独立的配置构建器：
- 原料池浏览（按源分组，支持搜索）
- 多套方案管理（CRUD）
- 深度编辑（逐字段修改 + 手动新增条目）
- 离线包导出（zip = config.json + jars/）

## 改动范围

### 1. 源追踪增强（聚合引擎 — 高风险区）

**文件：`src/core/merger.ts`**
- 当前：`siteSourceMap` 只追踪 sites
- 改动：新增 `parseSourceMap` 和 `liveSourceMap`，合并时为每个 parse/live 记录来源
- `MergeResult` 接口扩展：
  ```typescript
  interface MergeResult {
    config: TVBoxConfig;
    siteSourceMap: Map<string, string>;
    parseSourceMap: Map<string, string>;  // parse.url → sourceName
    liveSourceMap: Map<string, string>;   // live.url/api → sourceName
  }
  ```

**文件：`src/aggregator.ts`**
- 将新增的 source map 存入缓存（供 builder 端点读取）

**风险**：这是高风险区，但改动是纯"追加信息"——不改变现有合并/去重逻辑，只在旁边记录来源。影响可控。

### 2. 方案存储模块（新文件）

**新建：`src/core/builder-store.ts`**

Node.js only，用 JSON 文件存储（`data/presets/*.json`）：

```typescript
interface BuilderPreset {
  id: string;
  name: string;
  description?: string;
  createdAt: number;
  updatedAt: number;

  // 条目（自包含完整数据）
  sites: BuilderSite[];
  parses: BuilderParse[];
  lives: BuilderLive[];

  // 导出设置
  exportSettings: {
    path: string;           // 默认 /sdcard/TVBox/
    spiderStrategy: 'global' | 'per-site';  // 全局 spider vs 每站独立 jar
  };
}

// BuilderSite = TVBoxSite + 元数据
interface BuilderSite extends TVBoxSite {
  _source?: string;         // 从哪个源导入的（仅参考）
  _importedAt?: number;     // 导入时间
  _manual?: boolean;        // 是否手动添加
}
```

### 3. Builder API 端点（路由层 — 高风险区）

**文件：`src/routes.ts`（或拆出 `src/routes/builder.ts`）**

| 端点 | 方法 | 功能 |
|------|------|------|
| `/builder` | GET | 返回 Builder 页面 HTML |
| `/builder/pool` | GET | 返回原料池（含源分组信息） |
| `/builder/presets` | GET | 列出所有方案 |
| `/builder/presets` | POST | 创建方案 |
| `/builder/presets/:id` | GET | 获取单个方案 |
| `/builder/presets/:id` | PUT | 更新方案 |
| `/builder/presets/:id` | DELETE | 删除方案 |
| `/builder/presets/:id/export` | POST | 导出离线包（返回 zip） |

**风险**：路由是高风险区，但这些是纯新增端点，不改动现有路由逻辑。用 `app.route('/builder', builderRoutes)` 隔离。

### 4. 导出引擎（新文件）

**新建：`src/core/builder-export.ts`**

```
输入：BuilderPreset
输出：zip Buffer

流程：
1. 从 preset 收集所有 JAR 依赖（spider + 各站点 jar 字段）
2. 去重 JAR 列表
3. 逐个获取 JAR（优先 data/jars/ 缓存，miss 则下载）
4. 生成 config.json：
   - spider → file://{exportPath}/jars/{filename}.jar;md5;xxx
   - sites[].jar → file://{exportPath}/jars/{filename}.jar;md5;xxx
   - 保留所有用户编辑的字段
5. 用 JSZip (或 archiver) 打包
6. 返回 Buffer
```

**JAR 文件命名**：用原始 URL 的 SHA-256 短 key（复用 `jar-proxy.ts:urlToKey`），保持一致性。

### 5. Builder 页面（新文件）

**新建：`src/core/builder-ui.ts`**

页面布局（参考现有 admin/config-editor 风格）：

```
┌─────────────────────────────────────────────┐
│ Header: TVBox Config Builder                │
│ [方案选择下拉] [新建] [导出zip] [导入JSON]   │
├─────────────┬───────────────────────────────┤
│  原料池      │  当前方案                      │
│  (左侧面板)  │  (右侧面板)                    │
│             │                               │
│ ▼ 源A       │  Sites (12)                   │
│   □ 站点1   │  ┌─────────────────────────┐  │
│   □ 站点2   │  │ 站点X [编辑] [删除]      │  │
│ ▼ 源B       │  │ 站点Y [编辑] [删除]      │  │
│   □ 站点3   │  └─────────────────────────┘  │
│             │                               │
│ [手动添加]   │  Parses (5)  Lives (3)       │
│             │  ...                          │
├─────────────┴───────────────────────────────┤
│ 编辑面板 (点击"编辑"时展开)                    │
│ name: [____] api: [____] type: [v]          │
│ ext: [________________] jar: [____]         │
│ playerType: [v] searchable: [x]             │
│ [保存] [取消]                                │
└─────────────────────────────────────────────┘
```

核心交互：
- 左侧原料池：按源分组浏览，checkbox 多选 → "添加到方案"
- 右侧当前方案：已选条目列表，支持编辑/删除/拖拽排序
- 手动添加：弹出空白表单，填写后加入方案
- 编辑：点击条目展开字段编辑面板
- 导出：弹窗选路径 → 生成 zip 下载

### 6. 条件编译 / 入口隔离

**文件：`src/node-entry.ts`**
- Builder 路由只注册在 Node.js 入口
- CF Worker 入口（`src/cf-entry.ts`）不引入 builder 相关模块

## 不触及的区域

| 高风险区 | 是否受影响 | 说明 |
|----------|-----------|------|
| 聚合引擎 | 轻微 | 仅追加 source map，不改合并逻辑 |
| 路由层 | 新增 | 纯新端点，不改现有路由 |
| 存储层 | 不碰 | 方案用独立 JSON 文件，不走 Storage 抽象 |
| JAR 代理 | 不碰 | 复用 urlToKey + 缓存文件，不改代理逻辑 |
| 解码器 | 不碰 | 无关 |

## 风险点

1. **ZIP 生成的内存压力**：多个 JAR 可能 20-50MB，需要流式打包而非全部加载到内存
2. **JAR 下载超时**：某些 JAR CDN 可能慢/不可达，导出时需要处理下载失败
3. **源追踪改动对聚合性能的影响**：额外 Map 操作，但数据量小（几百条目），可忽略
4. **TVBox 路径兼容性**：不同 TVBox 分支对 `file://` 路径的处理可能有差异

## 实施阶段

| 阶段 | 内容 | 产出 |
|------|------|------|
| Phase 1 | 源追踪增强 + 方案存储 + API | 后端就绪，可用 curl 测试 |
| Phase 2 | Builder UI（浏览 + 选择 + 编辑） | 页面可用，方案可管理 |
| Phase 3 | 导出引擎 + ZIP 下载 | 完整功能闭环 |

## 验证方式

- Phase 1：`curl /builder/pool` 返回按源分组的条目；CRUD preset 正常
- Phase 2：浏览器操作 builder，能从池子添加/手动新增/编辑条目
- Phase 3：导出 zip，解压后 config.json 可被 TVBox 正常加载，JAR 文件完整可用

## 讨论记录

- 否决了"改造现有 config-editor"——职责不同，强行合并会混乱
- 否决了"CF Worker 也支持导出"——CPU/内存限制，且 CF 不需要此功能
- 方案存完整数据，不依赖原料池引用——解耦后不存在"过期"问题
- 源追踪需改动聚合引擎，但只追加信息，不改核心合并逻辑
- 允许手动添加条目——编辑器不仅是聚合池的定制层，也是完整的配置编写工具
