## 问题

TVBox 聚合器合并多个源时，不同源的 spider JAR 混在一起导致：
1. 搜索崩溃 — 去重后站点引用了错误 JAR 中不存在的类
2. JAR 源死亡 — 原始 URL 404/401 导致站点批量失效（实测 153 个站点已坏）
3. 碎片化 — 35 个不同 JAR，投票选全局 spider 结果不确定

## 根因分析

根本矛盾：每个源作者假设"我的 JAR + 我的站点是自洽的整体"，聚合器合并时打破了这个隐式耦合。

当前代码：
- `merger.ts` 用引用计数投票选全局 spider（不稳定）
- `dedup.ts` 去重键是 `key|api` 不含 jar（JAR 引用可能断裂）
- `jar-proxy.ts` 只转发 URL 不持久化 JAR（源死亡即失效）

## 方案

JAR 仓库模式：聚合器从"搬运工"升级为"仓库管理员"。

Phase 1: 聚合时下载每个唯一 JAR → 解析 DEX 提取类名清单 → 持久化存储
Phase 2: 对每个 type:3 csp_ 站点，检查类是否在某个 JAR 中存在 → 精确分配
Phase 3: orphaned 站点（类不存在）设 searchable: 0 防止搜索崩溃
Phase 4: CF 环境 base64 持久化 JAR 到 KV，源死亡时降级读取

功能开关默认关闭，不启用时行为与现有完全一致。

## 改动范围

- `src/core/types.ts`：新增 JarMeta/JarRegistry/JarAssignment 接口
- `src/core/config.ts`：新增 3 个 KV 键常量
- `src/core/dex-parser.ts`：**新建** DEX 字节码解析器
- `src/core/jar-registry.ts`：**新建** JAR 仓库核心模块
- `src/core/jar-proxy.ts`：urlToKey 改 export + base64 辅助函数
- `src/core/merger.ts`：mergeConfigs 新增可选 JarAssignment 参数
- `src/aggregator.ts`：Step 3.5 JAR 仓库分析
- `src/routes.ts`：4 个 admin API + /jar/:key 降级
- `src/core/admin.ts`：Settings 新增 JAR Registry 区块

## 风险点

- DEX 解析器 bug → orphaned 站点保留原样不删除 + 功能开关可关闭
- CF Worker CPU 限制 → 元数据缓存 + 冷启动阈值保护
- 多 JAR 含同名类 → 优先选站点原始源引用的 JAR

## 验证方式

1. 类型检查 `npx tsc --noEmit` 通过
2. 启用开关 → 刷新 → 查看报告确认覆盖率
3. 对比开启前后配置输出：orphaned 站点 searchable=0，per-site jar 正确
4. 关闭开关 → 行为与改动前一致

## 讨论记录

- 最初考虑"一 JAR 到底"（砍掉不兼容站点）→ 数据分析发现会损失 51% 站点，放弃
- 考虑合并 DEX 造超级 JAR → 类名冲突 + 维护成本高，放弃
- 最终选择"JAR 仓库"模式：不造新 JAR，而是管理已有 JAR 的分配关系
