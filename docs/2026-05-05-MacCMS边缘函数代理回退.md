## 问题

本地部署（Docker/Node.js）下，MacCMS 源（如玉兔资源站 `apiyutu.com`）
聚合器容器直连 SSL 被国内网络 reset，TVBox 访问 /api/:key 代理后
容器 fetch 原站失败，用户体验仍为"分类加载失败"。

上一轮方案已把 /api/:key 挂到本地模式，但 fetch 仍是容器直接出站，
没有利用项目已有的 edge proxy（CF Worker + Vercel）机制。

## 根因

- `src/routes.ts:737` 的 fetch 未读 `KV_EDGE_PROXIES`
- 对比 `src/aggregator.ts:134-150`，源 JSON 抓取有完整的 edge 回退链，
  MacCMS 代理这层漏接

## 现状验证（2026-05-05）

- NAS 容器 → `apiyutu.com` : SSL reset（出口不通）
- CF edge `tv.rio.edu.kg/fetch-proxy`（Bearer `qwer1234!`）→ 玉兔: 200 / 1.19s
- Vercel edge `fetch.riowang.win/api/proxy` → 玉兔: 200 / 0.88s ← 最快
- 存储中 `edge_proxies = {"cf":"https://tv.rio.edu.kg","vercel":"https://fetch.riowang.win"}` 已配置

## 方案

仅本地模式（`!config.workerBaseUrl`）下，/api/:key handler 读 `KV_EDGE_PROXIES`，
按「Vercel → CF → 直连」顺序尝试，首个成功即返回，全失败返回 502。

CF 版自身就在 edge 上，保持原直连逻辑不变。

每个 attempt 带 8s `AbortSignal.timeout(8000)`，避免单源拖死请求。
每次 attempt 打一行日志：`[maccms-proxy] <key> via <vercel|cf|direct> <ok|fail>: <status/msg>`。

### 候选请求链构造

```ts
if (!config.workerBaseUrl) {
  const edgeRaw = await storage.get(KV_EDGE_PROXIES);
  if (edgeRaw) {
    const edge: EdgeProxyConfig = JSON.parse(edgeRaw);
    const encoded = encodeURIComponent(targetUrl.toString());
    if (edge.vercel) attempts.push({
      label: 'vercel',
      url: `${edge.vercel.replace(/\/$/,'')}/api/proxy?url=${encoded}`,
      headers: {},
    });
    if (edge.cf) attempts.push({
      label: 'cf',
      url: `${edge.cf.replace(/\/$/,'')}/fetch-proxy?url=${encoded}`,
      headers: config.adminToken
        ? { Authorization: `Bearer ${config.adminToken}` } : {},
    });
  }
}
// 兜底：容器直连
attempts.push({ label: 'direct', url: targetUrl.toString(), headers: {} });

for (const { label, url, headers } of attempts) {
  try {
    const resp = await fetch(url, {
      headers: { 'User-Agent': 'okhttp/3.12.0', ...headers },
      signal: AbortSignal.timeout(8000),
    });
    if (!resp.ok) {
      lastError = `upstream ${resp.status}`;
      console.log(`[maccms-proxy] ${key} via ${label} fail: ${lastError}`);
      continue;
    }
    const data = await resp.json();
    console.log(`[maccms-proxy] ${key} via ${label} ok`);
    return c.json(data, 200, {
      'Access-Control-Allow-Origin': '*',
      'Cache-Control': 'public, max-age=300',
    });
  } catch (e) {
    lastError = e instanceof Error ? e.message : String(e);
    console.log(`[maccms-proxy] ${key} via ${label} fail: ${lastError}`);
  }
}
return c.json({ error: lastError || 'All proxies failed' }, 502);
```

## 改动范围

- `src/routes.ts` `/api/:key` handler：加 edge 回退链 + 导入 `KV_EDGE_PROXIES`、
  `EdgeProxyConfig` 类型 + 日志
- **不改**：`maccms.ts`、`aggregator.ts`、CF Worker 部署（只动本地模式路径）

## 风险点

1. **adminToken 跨端一致性**：CF Worker 的 ADMIN_TOKEN 和本地 Docker 的 ADMIN_TOKEN
   必须一致才能通过 CF edge 认证。当前测试证实 `qwer1234!` 两端一致。
   如未来分离，CF edge 会 401 → 自动 fallthrough 到直连或 Vercel，不会全盘崩。
2. **edge 返回非 JSON**（错误页/速率限制 HTML）：`.json()` 抛错 → 进入 catch
   尝试下一个。行为正确。
3. **回退顺序**：Vercel 优先（测试最快且无需 token）。如 Vercel 配额告急，
   手动在 admin 后台删除 vercel 字段即可跳过。
4. **POST 方法**：`/fetch-proxy` 和 `/api/proxy` 只支持 GET。若 MacCMS 站要求
   POST，全链路不通。本次不新增，标记已知限制。
5. **单 attempt 超时 8s**：能直连的源若 edge 两跳都失败再直连，最坏等 24s。
   可接受（MacCMS 实际调用通常 1-2s，失败极快）。

## 验证方式

1. `npx tsc --noEmit` 通过
2. NAS 部署后：
   `curl 'http://192.168.31.18:5678/api/yutuzy?ac=list' | head -c 200`
   应返回玉兔分类 JSON（`{"code":1,"msg":"数据列表","list":[...]}`）
3. 容器日志应看到：`[maccms-proxy] yutuzy via vercel ok`
4. 电视端：玉兔资源站分类/搜索正常
5. 回归：
   - CF 版部署不受影响（`workerBaseUrl` 短路，维持原分支）
   - 能直连的 MacCMS 源（如电影天堂 `dyttzyapi.com`）先走 vercel 也 OK，不会回归

## 讨论记录

- 放弃方案 B（api URL 直接写成 edge 地址）：
  - `/fetch-proxy` 只读 `?url=`，无法透传 TVBox 带的 `?ac=list&t=1`，
    要改 CF Worker 语义 + 重部署 = 高风险
  - 原站 URL 泄露在 TVBox 配置里，破坏"聚合器代理隐藏原站"的架构
  - CF auth token 没法让 TVBox 带
- 放弃方案 C（源级配置 useEdge 开关）：
  - 涉及前端 UI 改动，超出本次修复范围
  - 当前回退链（edge → 直连）已经自然覆盖两种场景
- 用户确认四项：方案 A / 回退顺序 Vercel→CF→直连 / 加日志 / 单 attempt 8s 超时
