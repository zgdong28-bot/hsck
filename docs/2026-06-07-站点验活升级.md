## 问题

现有站点测速（Step 6）仅检测 HTTP 可达性（TTFB），无法发现"HTTP 200 但内容无效"的死站（如返回空列表、返回错误页、API 格式变更）。用户大量聚合后，这类"假活"站点混入输出，TVBox 客户端表现为"能看到站点但点进去啥都没有"。

## 根因分析

`speedtest.ts:httpSpeedTest()` 只做 `fetch → resp.ok → 返回耗时`，不检查响应体内容。对 type0/type1 站点来说，CMS API 返回 200 但 `list` 为空（或返回 HTML 错误页）是常见的"假活"状态。

## 方案

升级现有 Step 6 测速逻辑，从"HTTP 可达"升级为"内容有效"验证：

### 验活逻辑（按 type 分）

| type | 探测 URL | 有效判定 |
|------|---------|---------|
| type0 (XML) | `api?ac=list` | 响应包含 `<list>` 或 `<video>` 标签 |
| type1 (JSON) | `api?ac=list` | 响应可解析为 JSON 且 `list` 数组长度 > 0，或 `class` 数组长度 > 0 |
| type3 (JAR) | api URL 本身 | 仅 HTTP 可达（JAR 站点无法 HTTP 验活） |
| type4/其他 | 不测 | 跳过（需要运行时，本阶段不支持） |

### 连续失败追踪

新增 KV key `site_health`，结构：

```typescript
interface SiteHealthRecord {
  key: string;           // site.key
  consecutiveFailures: number;
  lastProbeTime: string;
  lastProbeResult: 'ok' | 'empty' | 'error' | 'timeout';
  lastSuccessTime?: string;
}
type SiteHealthMap = Record<string, SiteHealthRecord>;
```

每次聚合 Step 6 更新：
- 验活通过 → `consecutiveFailures = 0`
- 验活失败 → `consecutiveFailures++`

### 自动标记 & 可选屏蔽

- `consecutiveFailures >= 3` 的站点：name 追加 `[⚠]` 标记
- `consecutiveFailures >= 5` 且"自动清理"开关打开：自动加入黑名单（指纹屏蔽）
- "自动清理"开关默认**关闭**（`KV_SITE_AUTO_CLEAN = 'site_auto_clean'`）
- 安全阀：单次聚合最多自动屏蔽 5 个站点（防止网络波动误杀一片）

### Admin 开关

- `验活深度`：shallow（仅 TTFB，现有行为）/ deep（检查内容有效性）
  - KV key: `site_probe_depth`，默认 `deep`
- `自动清理`：开/关
  - KV key: `site_auto_clean`，默认 `false`

## 改动范围

- `src/core/config.ts`：新增 `KV_SITE_HEALTH_MAP`、`KV_SITE_PROBE_DEPTH`、`KV_SITE_AUTO_CLEAN`
- `src/core/speedtest.ts`：
  - `httpSpeedTest()` → `siteProbe()`：升级验证逻辑，返回 `{ speedMs, result: 'ok'|'empty'|'error'|'timeout' }`
  - `getTestableUrl()` → 扩展为按 type 返回探测策略
  - 新增 `validateResponseContent(type, body): boolean`
- `src/aggregator.ts` Step 6：
  - 读取 `site_probe_depth` 决定走 shallow 还是 deep
  - 调用升级后的 `siteProbe`
  - 更新 `site_health` map
  - 根据 `consecutiveFailures` 追加标记或触发自动屏蔽
- `src/core/types.ts`：新增 `SiteHealthRecord`、`SiteHealthMap` 类型
- `src/routes.ts`：新增 admin API
  - `GET /admin/site-probe-depth` / `PUT /admin/site-probe-depth`
  - `GET /admin/site-auto-clean` / `PUT /admin/site-auto-clean`
  - `GET /admin/site-health` — 返回站点健康状态列表

## 风险点

1. **网络波动误杀**：单次网络抖动 → 多站同时失败 → 连续计数飙升。缓解：安全阀（单次最多屏蔽 5 个）+ 默认关闭自动清理
2. **验活耗时增加**：deep 模式每站要等响应 body（而非 HEAD），聚合总时间会增加。缓解：保持现有并发模式 + 超时时间不变（3s）
3. **type0 XML 判定误报**：有的 type0 站点返回的不是标准 XML，可能包含 `<list>` 但实际是错误页。缓解：同时检查 Content-Type 是否为 XML/text

## 验证方式

- 配置一个已知失效的 type1 站点（返回空 list），验证被标记为 `empty`
- 连续跑 3 次聚合，确认 `consecutiveFailures` 正确累加
- 开启自动清理，第 5 次聚合后确认站点被加入黑名单
- 关闭自动清理，确认即使 consecutiveFailures=10 也不屏蔽
- 确认 type3/type4 站点不受影响（跳过验活，保持现有 TTFB 逻辑）

## 讨论记录

- 最初提议独立 cron 模式（方案 A），但确认项目功能已较繁杂，选择轻量级方案 B
- type0 格式差异问题：实际 type0 和 type1 走同一套 CMS API 协议（只是响应格式不同），验活逻辑可以统一
- 播放链接验证（第五关）本阶段不做：防盗链导致服务端验证误判率高，收益不明确
- 方案从"五关"简化为"两关"（首页内容验证 + 可选搜索验证），足以区分"假活"站点
