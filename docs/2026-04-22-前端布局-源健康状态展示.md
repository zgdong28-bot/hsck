## 现状分析

### 页面结构

**Dashboard (`src/core/dashboard.ts`)**
- 总行数：468 行（单一导出模板字符串）
- 函数数量：5 个 JS 函数（`loadStatus`, `triggerRefresh`, `copyUrl`, `t`, `doToggleLang`）
- 最大函数行数：约 40 行（`loadStatus`）
- 布局结构：单列线性流——header > stats-grid(2x2) > update-section > config-section > footer
- CSS 方式：页面特有样式内联于 `<style>` 块，公共部分来自 `sharedStyles`

**Admin (`src/core/admin.ts`)**
- 总行数：889 行（单一导出模板字符串）
- 函数数量：约 20 个 JS 函数
- 最大函数行数：约 45 行（`loadSources`）
- 布局结构：login-overlay + container(header > tabs(4) > tab-panels > footer)
- 源列表采用 `.source-item` 组件：flex 行 = tag + info(name+url) + actions

### 问题诊断

Dashboard 当前只展示 4 个数字统计卡片 + 最后聚合时间，没有任何按源粒度的信息。用户看到"总站点数 120"但无法知道哪些源挂了。这是一个信息层级的断层：汇总有了，明细缺了。

Admin 源列表当前是纯静态展示（名称+URL+删除按钮），没有任何运行时状态反馈。用户添加了一个源之后，只能等下次聚合看总数有没有变，完全没有"这个源健不健康"的直觉。

### 操作动线问题

- Dashboard：用户想知道"哪个源出了问题"需要离开 dashboard 去看日志或手动测试，没有直接入口
- Admin：用户想判断"这个源该不该删"时，只能看 URL 猜，没有健康状态辅助决策

## 布局方案

### Dashboard：源健康状态区块

#### 变体 A：汇总条 + 可折叠表格（推荐）

**核心思路**：紧凑汇总放在 update-section 下方作为"第二信息层"，详情表格默认折叠，点击展开。
信息密度高但不压倒现有布局的简洁感。

**布局结构**：
```
┌─────────────────────────────────────┐
│  Header                             │
├─────────────────────────────────────┤
│  Stats Grid (2x2)                   │
├─────────────────────────────────────┤
│  Update Section (last aggregation)  │
├─────────────────────────────────────┤
│  Source Health Summary Bar          │  <-- 新增：N ok / N warn / N error
│  ┌─────────────────────────────────┐│
│  │  [展开] Health Detail Table     ││  <-- 默认折叠
│  │  ┌──┬────┬───┬───┬────┬──────┐ ││
│  │  │状│名称│URL│状态│失败│最后成功││
│  │  │态│    │   │    │次数│时间  ││
│  │  └──┴────┴───┴───┴────┴──────┘ ││
│  └─────────────────────────────────┘│
├─────────────────────────────────────┤
│  Config URL Section                 │
├─────────────────────────────────────┤
│  Footer                             │
└─────────────────────────────────────┘
```

**优势**：
- 默认折叠不增加页面长度，保持 dashboard 的"一屏概览"调性
- 汇总条一眼看出全局状态（全绿 = 不用展开）
- 利用现有的 `.collapsible-toggle` / `.collapsible-body` 组件，无需新增交互模式

**劣势**：
- 折叠意味着有问题时用户需要多点一次

#### 变体 B：汇总条 + 始终可见表格

**核心思路**：汇总条和表格始终展示，不折叠。适合源数量较少（<15）的场景。

**布局结构**：同变体 A，但表格始终可见。

**优势**：
- 零交互就能看到全部状态
- 有问题的源排在最前，红色行第一时间抓住注意力

**劣势**：
- 源多的时候（20+）页面会很长
- 破坏 dashboard 的简洁感

#### 变体 C：智能折叠（推荐的折中）

**核心思路**：汇总条始终可见。有报警（红色）时自动展开表格，全绿时默认折叠。

**优势**：
- 正常时保持简洁，出问题时自动暴露细节
- 符合"按异常驱动注意力"的监控原则

**劣势**：
- 实现多了一个条件判断，但很小

**最终推荐：变体 C（智能折叠）**。理由：dashboard 是监控页面，"没事别烦我，有事自动告诉我"符合操作直觉。

### Admin：源列表状态圆点

#### 方案：内联状态圆点 + 纯 CSS tooltip

**核心思路**：在现有 `.source-item` 的 `.source-info` 前面插入一个状态圆点（`.source-health-dot`），hover 时显示 tooltip。不改变现有布局结构。

**修改后的 source-item 结构**：
```html
<div class="source-item">
  <span class="source-health-dot ok"
        data-tooltip="Status: ok | Fails: 0 | Last OK: 2026-04-22 10:30">
  </span>
  <div class="source-info">
    <div class="source-name">某某源</div>
    <div class="source-url">https://...</div>
  </div>
  <div class="source-actions">
    <button class="btn btn-sm btn-danger">Remove</button>
  </div>
</div>
```

**状态颜色映射**：
- `.source-health-dot.ok` — `var(--green)` + glow
- `.source-health-dot.warn` — `var(--amber)` + glow
- `.source-health-dot.error` — `var(--red)` + glow
- `.source-health-dot.unknown` — `var(--text-dim)`，无 glow

**tooltip 实现**：纯 CSS `::after` 伪元素 + `data-tooltip` 属性。hover 时显示，不引入 JS tooltip 库。

**理由**：
- 圆点位置在 source-info 左侧，与现有 `.source-tag`（MacCMS 列表中的 key 标签）位置一致，视觉语言统一
- 纯 CSS tooltip 零依赖，符合项目约束
- 不影响现有 `.source-item` 的 flex 布局——只是多一个 flex-shrink:0 的小元素

## JS 逻辑设计

### Dashboard `loadSourceHealth()` 函数

```javascript
async function loadSourceHealth() {
  try {
    const res = await fetch('/admin/source-status');
    const records = await res.json();
    
    // 分类统计
    let ok = 0, warn = 0, err = 0;
    records.forEach(r => {
      if (r.consecutiveFailures >= 5) err++;
      else if (r.consecutiveFailures >= 3) warn++;
      else ok++;
    });
    
    $('healthOk').textContent = ok;
    $('healthWarn').textContent = warn;
    $('healthError').textContent = err;
    
    // 排序：失败次数降序
    records.sort((a, b) => b.consecutiveFailures - a.consecutiveFailures);
    
    // 渲染表格
    renderHealthTable(records);
    
    // 智能折叠：有 error 级别时自动展开
    if (err > 0) {
      $('healthToggle').classList.add('open');
      $('healthBody').classList.add('open');
    }
  } catch {
    $('healthTableBody').innerHTML =
      '<tr><td colspan="6" class="empty">' + t('fetchError') + '</td></tr>';
  }
}
```

### Dashboard `renderHealthTable(records)` 函数

```javascript
function renderHealthTable(records) {
  if (!records.length) {
    $('healthTableBody').innerHTML =
      '<tr><td colspan="6" class="empty">' + t('noData') + '</td></tr>';
    return;
  }
  
  $('healthTableBody').innerHTML = records.map(r => {
    const level = r.consecutiveFailures >= 5 ? 'error'
               : r.consecutiveFailures >= 3 ? 'warn' : 'ok';
    const statusLabel = {
      ok: 'OK', http_error: 'HTTP ERR', decode_error: 'DECODE ERR',
      parse_error: 'PARSE ERR', timeout: 'TIMEOUT', network_error: 'NET ERR'
    }[r.latestStatus] || r.latestStatus;
    
    const lastOk = r.lastSuccessTime
      ? new Date(r.lastSuccessTime).toLocaleString('zh-CN', {
          month:'2-digit', day:'2-digit', hour:'2-digit', minute:'2-digit', hour12:false
        })
      : '--';
    
    return `<tr class="row-${level}">
      <td><span class="health-dot ${level}"></span></td>
      <td>${esc(r.name || 'Unnamed')}</td>
      <td class="url-cell" title="${esc(r.url)}">${esc(r.url)}</td>
      <td class="status-${level}">${statusLabel}</td>
      <td>${r.consecutiveFailures}</td>
      <td>${lastOk}</td>
    </tr>`;
  }).join('');
}
```

### Admin `loadSourceHealth()` 函数

```javascript
let healthMap = {};

async function loadSourceHealth() {
  try {
    const res = await auth.authFetch('/admin/source-status');
    const records = await res.json();
    healthMap = {};
    records.forEach(r => { healthMap[r.url] = r; });
    // 重新渲染源列表以更新圆点
    loadSources();
  } catch {
    healthMap = {};
  }
}
```

调用时序：`loadAll()` 中先调 `loadSourceHealth()`，再调 `loadSources()`。或者 `loadSourceHealth` 拿到数据后自动触发 `loadSources` 重渲染。需要避免循环调用——`loadSources` 不再调 `loadSourceHealth`，两者独立。

推荐做法：`loadAll()` 中 `loadSourceHealth()` 只更新 `healthMap`，不触发 `loadSources()`。`loadSources()` 渲染时读 `healthMap`。两者在 `loadAll()` 中按顺序调用即可。

## 风险点

1. **API 端点认证差异**：Dashboard 的 `loadSourceHealth()` 直接 `fetch('/admin/source-status')`，Admin 的用 `auth.authFetch()`。如果 `/admin/source-status` 端点需要认证，Dashboard 端会 403。需要确认该端点是否需要 token，或者是否需要一个不带认证的公开版本（如 `/source-health`）。

2. **healthMap 竞态**：Admin 中 `loadSourceHealth()` 和 `loadSources()` 如果不保证顺序，可能出现 `loadSources()` 执行时 `healthMap` 还是空的。建议在 `loadAll()` 中用 `await` 保证顺序，或者接受首次渲染时圆点可能是 `unknown` 状态。

3. **大量源的表格性能**：如果源数量超过 50，Dashboard 表格 DOM 节点多。但 TVBox 聚合器场景下源数量一般在 5-30 之间，不是问题。

4. **tooltip 在移动端不可用**：纯 CSS tooltip 依赖 hover，触屏设备无法触发。对于 admin 页面（桌面优先场景）可接受。如需移动端支持，后续可改为 click 触发。

5. **i18n 同步**：两个页面新增了独立的翻译条目，需要保持中英文一致。

## 验证方式

1. 本地启动后访问 Dashboard（`/status`），确认汇总条正确显示绿/黄/红数量
2. 点击展开详情表格，确认按失败次数降序排列，URL 截断正常
3. 模拟有报警源时（consecutiveFailures >= 5），确认表格自动展开
4. 访问 Admin，确认源列表每个条目左侧有状态圆点，颜色正确
5. Hover 圆点确认 tooltip 显示完整信息
6. 切换亮色/暗色主题，确认所有新增元素颜色正常
7. 切换中/英文，确认翻译正确
8. 在 560px 视口下确认响应式布局正常
