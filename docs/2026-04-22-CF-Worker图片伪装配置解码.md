## 问题

CF Worker 聚合器无法处理两个使用"图片伪装"格式的 TVBox 配置源：
- `http://www.饭太硬.com/tv` → JPEG 图片伪装 + gzip 压缩（52 站点、7 直播）
- `http://ok213.top/tv` → 302 重定向到阿里 OSS 的 BMP 图片伪装（80 站点、13 解析、8 直播）

两个源都被 `fetchSingleConfig` 静默跳过，导致聚合结果缺失大量站点。

## 根因分析

TVBox 生态中存在三种配置编码格式，用于反检测和防止被直接抓取：

### 格式 1：图片伪装 + Base64（两个失败 URL 都用此格式）
- 响应是合法的 JPEG/BMP 图片文件
- 图片数据后追加 `[A-Za-z0]{8}**` 标记（如 `TswYJXMV**`、`QhHIWSIv**`）
- 标记之后是 base64 编码的 JSON 配置
- TVBox 安卓端在 `ApiConfig.FindResult()` (ApiConfig.java:113-139) 中解码

### 格式 2：AES CBC 自解密（`2423` 前缀）
- 响应是十六进制字符串，以 `2423`（`$#` 的 hex）开头
- key 和 iv 嵌在数据内部：
  - 整体 hex → 字符串后，`$#` 和 `#$` 之间是 key（右补 `0` 到 16 位）
  - 最后 13 个字符是 iv（右补 `0` 到 16 位）
  - `#$` 之后到倒数第 26 字符是密文（hex 编码）
- 解密算法：AES/CBC/PKCS7Padding
- TVBox 安卓端在 `AES.CBC()` (AES.java:44-55) 中解密

### 格式 3：AES ECB 外部密钥
- 响应是十六进制字符串（hex 编码的密文）
- 密钥通过 URL 后缀 `;pk;密钥` 提供
- 解密算法：AES/ECB/PKCS7Padding，密钥右补 `0` 到 16 位
- TVBox 安卓端在 `AES.ECB()` (AES.java:30-42) 中解密

### CF Worker 当前逻辑（缺失解码）

```
fetchSingleConfig:
  fetch(url) → response.text() → parseConfigJson(text) → JSON.parse()
```

- `response.text()` 对二进制图片数据返回乱码
- `JSON.parse()` 对乱码/图片/hex 字符串一律失败
- 失败后返回 null → 该源被静默跳过

### IDN 域名问题（非问题）
CF Worker 和 Node.js 的 `fetch()` 都使用 WHATWG URL 解析，自动将 `www.饭太硬.com` 转为 Punycode `www.xn--sss604efuw.com`。`400 Bad Request` 仅在 curl 等不自动转换的工具中出现。

## 方案

新建独立解码模块 `decoder.ts`，完整移植 TVBox 安卓端的 `FindResult` + `AES` 解码能力。

### 架构

```
fetchSingleConfig (fetcher.ts)
  │
  ├─ fetch(url)
  ├─ response.arrayBuffer()  ← 改动：从 text() 改为 arrayBuffer()
  ├─ decodeConfigResponse(buffer, configKey?)  ← 新增：调用解码器
  │   ├─ 转 Latin-1 文本
  │   ├─ 检查是否为 JSON → 直接返回
  │   ├─ 检测 [A-Za-z0]{8}** → 提取 base64 → 解码
  │   ├─ 检测 2423 开头 → AES CBC 自解密
  │   ├─ 有 configKey → AES ECB 解密
  │   └─ 返回解码后的文本（或 null）
  ├─ parseConfigJson(decodedText)  ← 现有逻辑不变
  └─ return SourcedConfig
```

### 改动范围

#### 1. 新建 `scripts/cf-worker/src/core/decoder.ts`
- `decodeConfigResponse(buffer: ArrayBuffer, configKey?: string): string | null`
  - 主入口，按优先级尝试各种解码
- `decodeImageWrapped(text: string): string | null`
  - 正则 `/[A-Za-z0]{8}\*\*/` 检测标记
  - 跳过标记（10 字符），提取剩余部分
  - Base64 解码（使用 `atob()` 或 `Buffer.from(..., 'base64')`）
- `decryptAesCbc(hexContent: string): string | null`
  - 检测 `2423` 前缀
  - 从 hex 字符串中提取 key、iv、ciphertext
  - 使用 Web Crypto API (`crypto.subtle.decrypt`) 执行 AES-CBC 解密
- `decryptAesEcb(hexContent: string, key: string): string | null`
  - hex → bytes 转换
  - AES-ECB 解密（Web Crypto API 不直接支持 ECB，用 CBC + 逐块处理模拟）
- `hexToBytes(hex: string): Uint8Array` — 工具函数
- `rightPadding(str: string, pad: string, length: number): string` — 移植自 AES.java

### AES ECB 实现策略

Web Crypto API 不支持 ECB 模式。跨平台方案：

```typescript
// ECB = CBC with zero IV, per-block processing
async function aesEcbDecrypt(data: Uint8Array, key: Uint8Array): Promise<Uint8Array> {
  const cryptoKey = await crypto.subtle.importKey('raw', key, 'AES-CBC', false, ['decrypt']);
  const blocks: Uint8Array[] = [];
  const zeroIv = new Uint8Array(16);
  
  for (let i = 0; i < data.length; i += 16) {
    const block = data.slice(i, i + 16);
    const decrypted = await crypto.subtle.decrypt(
      { name: 'AES-CBC', iv: zeroIv },
      cryptoKey,
      block
    );
    blocks.push(new Uint8Array(decrypted));
  }
  // 手动去 PKCS7 padding
  return concat(blocks);
}
```

注意：这里需要用 `{ name: 'AES-CBC', iv: zeroIV }` 模拟 ECB，因为 ECB 每个块独立加密，等价于 CBC 模式下 IV 全零且不链接。

**但实际上不完全等价**——CBC 会把前一块密文作为下一块的 IV。真正模拟 ECB 需要每 16 字节独立解密一个块。上面的逐块方案就是正确的做法。

最后一块可能需要 PKCS7 去填充（CF Worker 的 decrypt 会自动去除）。

## 风险点

1. **`response.arrayBuffer()` 性能**：图片响应通常 20-50KB，不大。但比 `response.text()` 多一次 ArrayBuffer → 字符串转换。影响可忽略。
2. **Latin-1 编码转换**：二进制图片数据必须用 Latin-1（ISO-8859-1）转字符串，不能用 UTF-8（会丢失字节）。CF Worker 和 Node.js 都支持 `TextDecoder('latin1')`。
3. **AES ECB 逐块解密性能**：每 16 字节一次 `crypto.subtle.decrypt` 调用，对大数据量可能慢。但配置 hex 字符串一般几十 KB，对应密文只有几 KB，影响可忽略。
4. **base64 解码兼容性**：CF Worker 有 `atob()`，Node.js 用 `Buffer.from(str, 'base64')`。需要写兼容代码或使用 polyfill。
5. **configKey 存储安全**：密钥以明文存储在 KV 中。如果 admin API 有 token 保护，风险可控。
6. **向后兼容**：`SourceEntry` 新增 `configKey` 是可选字段，现有数据不受影响。

## 验证方式

1. **单元验证**：用两个已知 URL 的响应数据（可以 curl 保存为文件）测试 `decodeConfigResponse()`
   - ok213.top 的 BMP 响应 → 应解出 80 个站点
   - 饭太硬.com 的 JPEG 响应 → 应解出 52 个站点
2. **集成验证**：本地启动 Node.js 版 CF Worker，添加这两个源后触发聚合，检查聚合结果中的站点数量
3. **回归验证**：确认现有纯 JSON 格式的源不受影响（`decodeConfigResponse` 对纯 JSON 直接透传）

## 讨论记录

### 被考虑但未采用的方案
- **方案 A（仅 base64 解码）**：只解决当前两个 URL 但不支持加密格式，扩展性差
- **在 parseConfigJson 中扩展**：职责不清晰，parseConfigJson 负责 JSON 容错解析，不应承担二进制解码

### 关键发现
- `ok213.top/tv` 的格式：BMP 文件头（1034 bytes）+ `TswYJXMV**` 标记 + base64 JSON
- `饭太硬.com/tv` 的格式：JPEG 图片（gzip 压缩传输）+ `QhHIWSIv**` 标记 + base64 JSON
- 两者都匹配 TVBox 安卓端 `FindResult()` 的 `[A-Za-z0]{8}**` 正则
- CF Worker `fetch()` 自动处理 IDN Punycode 转换和 gzip 解压
