## 问题

当前聚合流程中，源 fetch 失败只打 `console.warn` 然后静默跳过，失败信息零持久化。管理员无法知道哪些源挂了、挂了多久、失败原因是什么。只能等到聚合输出为空才发现所有源都不可用——发现太晚，损失太大。

## 根因分析

`fetcher.ts` 的 `fetchSingleConfig` 失败时返回 `null`，在 `expandSources` 中被过滤掉，`aggregator.ts` 只拿到成功的 `SourcedConfig[]`，失败信息在 `Promise.allSettled` 处理后彻底丢失。

## 方案

**累积追踪模式**：在聚合流程内收集每个源的 fetch 结果，与历史状态 merge 后持久化，Dashboard 和 Admin 分级展示。

### 数据模型

```typescript
// 单次 fetch 结果（内部传递，不持久化）
interface SourceFetchResult {
  url: string;
  name: string;
  status: 'ok' | 'http_error' | 'decode_error' | 'parse_error' | 'timeout' | 'network_error';
  errorMessage?: string;
  speedMs?: number;
}

// 持久化的健康记录
interface SourceHealthRecord {
  url: string;
  name: string;
  latestStatus: 'ok' | 'http_error' | 'decode_error' | 'parse_error' | 'timeout' | 'network_error';
  consecutiveFailures: number;   // 连续失败次数，成功后归零
  lastSuccessTime?: string;      // ISO 时间戳
  lastFailTime?: string;
  lastFailReason?: string;
  lastSpeedMs?: number;          // 最近一次成功的响应耗时
}
```

### 严重程度阈值

| 级别 | 条件 | 颜色 | 含义 |
|------|------|------|------|
| 正常 | consecutiveFailures = 0 | 绿色 | 最近一次聚合成功 |
| 警告 | consecutiveFailures ≥ 3 | 黄色 | 连续多次失败，可能有问题 |
| 报警 | consecutiveFailures ≥ 5 | 红色 | 高概率已失效，建议替换 |
| 无数据 | 无记录 | 灰色 | 新增源，尚未聚合过 |

### Merge 逻辑

```
对每个源 URL：
  本次成功 → consecutiveFailures = 0, lastSuccessTime = now, lastSpeedMs = speedMs
  本次失败 → consecutiveFailures++, lastFailTime = now, lastFailReason = reason
  源已删除（不在当前源列表中）→ 从记录中移除
  新增源（无历史记录）→ 新建记录
```

## 改动范围

### 1. `src/core/types.ts`：新增类型
- 新增 `SourceFetchResult` 接口
- 新增 `SourceHealthRecord` 接口

### 2. `src/core/config.ts`：新增常量
- 新增 `KV_SOURCE_HEALTH = 'source_health'`

### 3. `src/core/fetcher.ts`：改造返回值
- `fetchConfigs` 返回值从 `SourcedConfig[]` 改为 `{ configs: SourcedConfig[], fetchResults: SourceFetchResult[] }`
- `fetchSingleConfig` 失败时返回结构化的 `SourceFetchResult`（含失败分类和原因），不再返回 `null`
- `expandSources` 同步收集 fetchResults
- 多仓子源的 fetch 结果也要收集

### 4. `src/aggregator.ts`：收集状态并持久化
- 接收 `fetchResults` 
- 读取历史 `KV_SOURCE_HEALTH` → merge 本次结果 → 写回
- merge 函数独立抽取，逻辑清晰

### 5. `src/routes.ts`：新增 API 端点
- `GET /admin/source-status`：返回 `SourceHealthRecord[]`
- 不需要 admin token 认证（Dashboard 需要无认证访问）
- 或设计为 `/source-status`（与 `/status-data` 同级，无认证）

### 6. `src/core/dashboard.ts`：新增源健康区块
- **位置**：update-section 和 config-section 之间
- **布局**：智能折叠——汇总条常驻（N正常/N警告/N报警），有报警时自动展开表格
- **表格列**：状态圆点、名称、URL（截断）、连续失败次数、最后成功时间、失败原因
- **排序**：失败的排前面（按 consecutiveFailures 降序）

### 7. `src/core/admin.ts`：源列表加状态图标
- 源列表每个条目左侧加 8px 状态圆点（绿/黄/红/灰）
- 纯 CSS tooltip（hover 显示详情：状态、连续失败次数、最后成功时间）
- 页面加载时异步 fetch `/admin/source-status` 获取状态数据

## 风险点

1. **API 认证策略**：Dashboard 页面不要求登录，`/admin/source-status` 如果走 admin token 认证会导致 Dashboard 403。建议该端点放到 `/source-status`（与 `/status-data` 同级），或 Dashboard 通过 `/status-data` 一起返回健康数据。
2. **KV 存储大小**：每个源一条记录，100个源约 10KB JSON，远低于 KV 25MB 限制，不是问题。
3. **fetcher 返回值变更**：`fetchConfigs` 签名变了，所有调用方需要适配。当前只有 `aggregator.ts` 一处调用，影响可控。

## 验证方式

1. 添加一个故意写错的源 URL → 聚合后 Dashboard 能看到该源标红 + 失败原因
2. 连续聚合 3 次 → 该源 consecutiveFailures 从 1 增长到 3，颜色从绿变黄
3. 修正 URL 后聚合 → consecutiveFailures 归零，颜色变绿
4. 删除源后聚合 → 健康记录中该源被自动清理
5. Admin 源列表中状态圆点与 Dashboard 一致

## 讨论记录

- 方案 A（快照模式，只看最近一次）被否决：无法区分临时故障和永久失效，误报率高
- 方案 B（累积追踪）确认：额外复杂度极低（~20行 merge 逻辑），信息量显著更大
- 展示位置：Dashboard + Admin 都做
- 阈值：3次黄色警告、5次红色报警
