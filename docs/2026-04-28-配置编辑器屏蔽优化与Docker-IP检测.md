## 问题

### 问题 1：配置编辑器屏蔽体验差
- 每次屏蔽一个站点/解析/直播后，折叠菜单自动收起，需要反复展开才能继续操作
- 没有批量屏蔽功能，大量站点需要逐个操作

### 问题 2：Docker 部署 IP 地址不正确
- 未配置 `BASE_URL` 时，`getLocalIp()` 获取到 Docker 容器内部 IP（172.17.x.x）
- 该 IP 被用于 JAR 代理地址改写，导致 TVBox 客户端无法访问
- 用户不知道需要配置 `BASE_URL`（`.env.example` 无此项）

## 根因分析

### 问题 1
`config-editor.ts:549-572` 的 `block()` 函数屏蔽成功后调用 `renderSites()` / `renderParses()` / `renderLives()`，这些函数用 `container.innerHTML = html` 全量重建 DOM，销毁所有 `.open` 折叠状态。折叠状态仅存于 DOM class，无持久化机制。

### 问题 2
`node-entry.ts:179-189` 的 `getLocalIp()` 用 `os.networkInterfaces()` 取第一个非 internal 的 IPv4 地址。Docker 容器内 `eth0` 为 172.17.x.x（`internal: false`），被错误选中。该值通过 `buildConfig()` → `config.localBaseUrl` → `aggregator.ts:259,301` 传入 JAR 代理地址改写链路。

## 方案

### 问题 1：局部 DOM 更新 + 批量多选屏蔽

#### 1.1 屏蔽/恢复改为局部 DOM 更新

改造 `block()` 和 `unblock()` 函数，成功后不再调用全量 `renderXxx()`，而是：
- 找到当前操作的 `.item` DOM 元素（通过 `data-id` 属性定位）
- 直接修改 class（添加/移除 `blocked`）
- 替换按钮文字和 onclick

#### 1.2 新增多选 + 批量屏蔽 UI

**前端**：
- 每个 `.item` 行左侧添加 checkbox（仅非 blocked 项显示）
- 分组标题添加"全选"checkbox
- 顶部工具栏添加"批量屏蔽"按钮（选中数量 > 0 时显示）
- 批量操作调用一次批量 API，成功后逐个更新 DOM

**后端**：
- `routes.ts` 新增 `POST /admin/blacklist/batch` 端点
- 接受 `{ type: string, ids: string[] }` 请求体
- 复用 `loadBlacklist` / `saveBlacklist`，一次性写入

### 问题 2：Docker 环境检测 + 警告

#### 2.1 Docker 环境检测

`node-entry.ts` 新增 `isDocker()` 检测函数：
- 检查 `/.dockerenv` 文件是否存在
- 或检查 `/proc/1/cgroup` 中是否含 `docker` / `containerd`

#### 2.2 IP 获取优化

Docker 环境下且未配置 `BASE_URL` 时：
1. 先尝试 DNS 解析 `host.docker.internal`（Docker Desktop 支持）
2. 成功 → 使用该 IP 构建 baseUrl
3. 失败 → `localBaseUrl` 仍使用容器 IP，但标记 `dockerMissingBaseUrl: true`

#### 2.3 启动日志警告

检测到 Docker 且未配置 `BASE_URL` 时，打印醒目警告：
```
⚠️  检测到 Docker 环境但未配置 BASE_URL
    当前地址 172.17.x.x 为容器内部 IP，TVBox 客户端可能无法访问
    请在 .env 或 docker-compose.yml 中设置：BASE_URL=http://宿主机IP:端口
```

#### 2.4 `/status-data` API 返回警告

`routes.ts` 的 `/status-data` 响应新增 `warnings: string[]` 字段。
当 `config.dockerMissingBaseUrl` 为 true 时，推入警告消息。

#### 2.5 Status 页面展示警告 Banner

`dashboard.ts` 在统计卡片上方（`.stats-grid` 之前）插入 `<div id="warningBanner">` 容器。
`loadStatus()` 解析 `warnings` 数组，有内容时渲染 amber 色 banner：

```html
<div class="warning-banner">
  ⚠ 检测到 Docker 环境但未配置 BASE_URL，JAR 代理地址可能不可达。
  请在 docker-compose.yml 中设置 BASE_URL=http://宿主机IP:端口
</div>
```

样式复用 `--amber` / `--amber-dim` 颜色变量。支持 i18n 中英文。

#### 2.6 配置文件补充

- `.env.example` 添加 `BASE_URL` 及注释说明
- `docker-compose.yml` 在 `BASE_URL` 行添加注释

## 风险点

- **问题 1**：局部 DOM 更新需要确保 `data-id` 唯一性（sites 用 fingerprint，parses 用 url，lives 用 url——与现有 block/unblock 一致，无新风险）
- **问题 2**：`host.docker.internal` 在 Linux Docker（非 Desktop）默认不可用，DNS 解析会失败——方案已考虑 fallback，不阻塞启动
- **问题 2**：`/.dockerenv` 检测方式在某些 CI 环境可能误报——影响仅为多一条日志警告，可接受

## 验证方式

### 问题 1
1. 打开配置编辑器，展开某个分组
2. 屏蔽其中一个站点 → 验证分组保持展开
3. 连续屏蔽多个站点 → 验证分组始终保持展开
4. 勾选多个站点，点击批量屏蔽 → 验证一次性全部屏蔽成功
5. 搜索过滤后进行屏蔽操作 → 验证搜索状态不丢失

### 问题 2
1. Docker 部署不配 BASE_URL → 验证启动日志有警告 + status 页面显示 amber banner
2. Docker 部署配置 BASE_URL → 验证无警告、JAR 代理地址正确
3. 非 Docker 环境部署 → 验证无警告（不误报）

## 讨论记录

- 方案 A（保存/恢复折叠状态）被否决：治标不治本，仍然全量重渲染
- 方案 C（引入 VDOM）被否决：过度工程化
- Docker IP 方案 A（过滤 Docker 网段）被否决：用户局域网可能使用相同网段，误杀风险大
- Docker IP 方案 C（仅用 host.docker.internal）作为补充探测手段保留，非主方案
