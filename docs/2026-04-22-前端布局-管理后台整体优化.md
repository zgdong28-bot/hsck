## 现状分析

### 页面结构总览

| 文件 | 行数 | 角色 | 结构特征 |
|------|------|------|---------|
| `src/core/admin.ts` | 1294 | 管理后台 | 单个 export，HTML+CSS+JS 全内联，**10 个平铺 section** |
| `src/core/dashboard.ts` | 726 | 状态仪表盘 | 单个 export，结构清晰，4 卡片 + 状态栏 |
| `src/core/config-editor.ts` | 1006 | 配置编辑器 | 单个 export，**已有 Tab 导航**，结构最优 |

总计 **3026 行**，其中约 **900 行是跨文件的重复代码**。

### 问题诊断

**1. admin.ts：10 个 section 平铺，无层级**

当前 admin 页面的信息架构：

```
Header
├── Import Config          ← 导入操作（低频）
├── Add Source             ← 添加表单
├── Aggregation            ← 刷新按钮（单行占一个 section）
├── Source List             ← 源列表
├── Add MacCMS             ← MacCMS 添加表单
├── MacCMS List            ← MacCMS 列表
├── Add Live Source        ← 直播源添加表单
├── Live Source List       ← 直播源列表
├── Name Transform         ← 名称定制设置（低频）
└── Footer
```

问题根因：每个新功能都作为独立 section 追加到底部，功能平权，没有分组。

**2. 跨文件 CSS 重复 ~700 行**

| 重复内容 | 出现次数 | 每份行数 | 浪费 |
|---------|---------|---------|-----|
| `:root` CSS 变量 | ×3 | ~30 | ~60 |
| `body` + 伪元素（网格/扫描线） | ×3 | ~30 | ~60 |
| `.header` 系列样式 | ×3 | ~60 | ~120 |
| `.lang-toggle` | ×3 | ~15 | ~30 |
| `.footer` | ×3 | ~10 | ~20 |
| 动画 keyframes | ×3 | ~25 | ~50 |
| `.login-overlay` + `.login-box` | ×2 | ~60 | ~60 |
| `.btn` 系列样式 | ×2 | ~40 | ~40 |
| **小计** | | | **~440** |

**3. 跨文件 JS 重复 ~200 行**

| 重复内容 | 出现次数 |
|---------|---------|
| i18n 系统（`getLang` / `t` / `applyLang` / `toggleLang`） | ×3 |
| Auth 逻辑（login overlay + sessionStorage） | ×2 |
| `esc()` 转义函数 | ×2 |

**4. 操作动线问题（admin.ts）**

- 添加一个 TVBox 源后，要向下滚动 3 屏才能看到源列表确认是否添加成功
- "Import Config"（低频操作）占据页面最顶部位置，比高频的"源列表"更显眼
- "Aggregation" 只有一个刷新按钮 + 一行状态文字，却独占一整个 section 卡片
- "Name Transform"（设置类，一次配好几乎不改）和数据管理操作混在一起

## 布局方案

### 变体 A：Tab 导航（推荐）

**核心思路**：借鉴 config-editor 已有的 Tab 模式，将 admin 页面的 10 个平铺 section 收纳为 4 个 Tab。聚合刷新上提到 header 区域。

**布局结构**：

```
┌──────────────────────────────────────────┐
│  Header                                  │
│  TVBox Source Manager                    │
│  [Admin] [Config Editor] [Dashboard]     │
│                                          │
│  Last update: 2026-04-22 10:30:00        │
│  128 sites, 12 parses     [Refresh]      │
├──────────────────────────────────────────┤
│  [Sources ·8] [MacCMS ·3] [Live ·2] [Settings] │
├──────────────────────────────────────────┤
│                                          │
│  ┌─ Add Source ─────────────────────┐    │
│  │ [Name] [URL] [Key]     [Add]    │    │
│  │ [Import Config ▾]               │    │
│  └──────────────────────────────────┘    │
│                                          │
│  ┌─ Source List ────────────────────┐    │
│  │  source-1           [Remove]    │    │
│  │  source-2           [Remove]    │    │
│  │  source-3           [Remove]    │    │
│  └──────────────────────────────────┘    │
│                                          │
│  Footer                                  │
└──────────────────────────────────────────┘
```

**Tab 分组**：

| Tab | 包含内容 | 操作动线 |
|-----|---------|---------|
| **Sources** | Add Source 表单 + Import（折叠）+ 源列表 | 添加 → 立即在下方看到结果 |
| **MacCMS** | Add MacCMS 表单 + Batch Import（折叠）+ MacCMS 列表 | 同上 |
| **Live** | Add Live Source 表单 + 直播源列表 | 同上 |
| **Settings** | Name Transform + 未来的其他设置 | 低频操作隔离 |

**Aggregation/Refresh 处理**：从独立 section 移到 header 下方的状态栏，和 dashboard 的交互模式统一。

**优势**：
- 信息分层清晰，一屏内完成"添加 → 确认"的完整动线
- 与 config-editor 的 Tab 交互一致，用户无学习成本
- 低频操作（Import、Settings）不干扰高频操作
- Tab badge 显示数量，快速感知各类源的状态

**劣势**：
- 需要重写 admin.ts 的 HTML 结构和 JS 交互
- Tab 切换时的状态保持需要额外处理（当前是纯前端渲染，问题不大）

---

### 变体 B：可折叠分组

**核心思路**：不引入 Tab，而是将相关的 section 合并为可折叠的分组（类似 config-editor 的 group），默认展开最常用的，折叠其余。

**布局结构**：

```
┌──────────────────────────────────────────┐
│  Header + Status Bar + [Refresh]         │
├──────────────────────────────────────────┤
│                                          │
│  ▼ TVBox Sources (8)                     │
│  ┌──────────────────────────────────┐    │
│  │ [Add form]                       │    │
│  │ source-1 / source-2 / ...       │    │
│  └──────────────────────────────────┘    │
│                                          │
│  ▶ MacCMS Sources (3)     [collapsed]    │
│                                          │
│  ▶ Live Sources (2)       [collapsed]    │
│                                          │
│  ▶ Settings               [collapsed]    │
│                                          │
│  Footer                                  │
└──────────────────────────────────────────┘
```

**优势**：
- 改动量最小，只需合并 section + 加折叠逻辑
- 所有内容仍在一个页面，可用 Ctrl+F 搜索

**劣势**：
- 折叠状态下仍然是垂直堆叠，滚动量只是从"必须"变成"可选"
- 多个分组同时展开时回到老问题
- 不如 Tab 干净，视觉噪音更多

---

### 变体 C：侧边导航 + 内容区

**核心思路**：左侧固定导航列出所有功能类别，右侧显示对应内容。

```
┌───────────┬──────────────────────────────┐
│  Logo     │  Header                      │
├───────────┤                              │
│           │                              │
│  Sources  │  [当前选中的功能区内容]        │
│  MacCMS   │                              │
│  Live     │                              │
│  Settings │                              │
│           │                              │
│  ──────── │                              │
│  Dashboard│                              │
│  Config   │                              │
│  Editor   │                              │
├───────────┴──────────────────────────────┤
│  Footer                                  │
└──────────────────────────────────────────┘
```

**优势**：
- 导航永远可见，切换零成本
- 可以把跨页面的导航（Dashboard、Config Editor）也收进来，统一入口
- 扩展性最好，未来加功能只需加菜单项

**劣势**：
- 改动量最大，三个页面都需要引入侧边栏
- 移动端需要 hamburger 菜单，额外复杂度
- 管理后台只有 4 个功能分组，侧边栏有点"大材小用"

---

### 建议

**推荐变体 A（Tab 导航）**。理由：

1. config-editor 已经验证了 Tab 模式在这个项目中好用
2. 改动集中在 admin.ts，不影响 dashboard 和 config-editor
3. 平衡了信息层级优化和实现复杂度
4. 移动端友好——Tab 可以水平滚动

变体 C 的侧边导航更适合功能分组超过 6-8 个时引入，当前阶段过度设计。

## Admin 页面具体改动（变体 A 落地）

### HTML 结构变更

**Before**（10 个平铺 section）：
```
container
  header
  section: Import Config
  section: Add Source
  section: Aggregation
  section: Source List
  section: Add MacCMS
  section: MacCMS List
  section: Add Live Source
  section: Live Source List
  section: Name Transform
  footer
```

**After**（Header 状态栏 + 4 Tab）：
```
container
  header
    nav links
    status-bar: "Last update ... | 128 sites ..." + [Refresh] button
  tabs: [Sources ·N] [MacCMS ·N] [Live ·N] [Settings]
  search-bar (仅 Sources/MacCMS/Live Tab 显示)
  tab-panel: Sources
    add-form (inline, URL + Name + Key + [Add])
    import-toggle ("Import Config ▾" 折叠区)
    source-list
  tab-panel: MacCMS
    add-form (inline, Key + Name + API + [Add])
    batch-toggle ("Batch Import ▾" 折叠区)
    maccms-list
  tab-panel: Live
    add-form (inline, Name + URL + [Add])
    live-list
  tab-panel: Settings
    name-transform (prefix / suffix / promo / patterns)
    (future: 其他设置)
  footer
```

### 交互变更

| 操作 | 现状 | 优化后 |
|------|------|-------|
| 添加 TVBox 源 | 填表 → 滚动 3 屏看列表 | 填表 → 列表就在下方 |
| 导入配置 | 页面顶部大 textarea | 折叠在 Add Source 下方，点击展开 |
| MacCMS 批量导入 | 显示/隐藏 textarea | 折叠，交互不变 |
| 刷新聚合 | 独占一个 section | Header 状态栏内的按钮 |
| 名称定制 | 和数据操作混排 | 独立 Settings Tab |
| 跨类型查看 | 全部可见需滚动 | Tab 切换，零滚动 |

### Tab 复用 config-editor 的 CSS

config-editor 已有的 `.tabs` / `.tab` / `.tab-panel` 样式直接提取到 `shared-styles.ts`，admin 复用。

## CSS 策略

### 不新增 CSS 变量

现有变量体系完全够用：
- 颜色：`--green`, `--amber`, `--red`, `--blue` + 各自 `dim` 变体
- 表面：`--bg`, `--surface`, `--surface-2`
- 边框：`--border`, `--border-glow`
- 文字：`--text`, `--text-dim`
- 字体：`--mono`, `--sans`

### 新增的 CSS class（提取到 shared-styles）

| Class | 来源 | 用途 |
|-------|------|------|
| `.tabs`, `.tab`, `.tab.active` | config-editor | Tab 导航 |
| `.tab-panel`, `.tab-panel.active` | config-editor | Tab 内容面板 |
| `.search-bar` | config-editor | 搜索栏 |
| `.status-bar` | dashboard | Header 状态栏 |
| `.section` | admin | 卡片容器 |
| `.source-list`, `.source-item` | admin | 源列表 |
| `.add-form` | admin | 添加表单 |
| `.collapsible-toggle` | **新增** | 折叠触发器（Import/Batch） |

### 响应式断点

- `≤560px`：Tab 水平滚动，add-form 换行堆叠，状态栏垂直排列
- `≤768px`：单列布局（已有，维持）
- `>768px`：完整布局

## 风险点

| 风险 | 影响 | 缓解 |
|------|------|------|
| 公共模块提取后，页面级 CSS 选择器冲突 | 样式错乱 | 每页检查特有 class 命名无冲突 |
| Tab 切换打断用户正在填写的表单 | 数据丢失 | Tab 切换不销毁 DOM，用 `display:none` 隐藏 |
| admin.ts HTML 重构后遗漏功能 | 功能缺失 | 逐一对照现有 10 个 section 的功能清单 |
| CF Worker bundle size 变化 | 部署限制 | 公共模块是 TS 常量，编译后会 tree-shake |

## 验证方式

1. `npm run dev:cf` 启动本地 Worker，逐一访问 `/admin`、`/status`、`/admin/config-editor`
2. Admin 页面：逐一测试 Sources/MacCMS/Live/Settings 四个 Tab 的 CRUD 操作
3. 切换中英文，确认所有 i18n 文本正确
4. 移动端模拟（Chrome DevTools），确认 ≤560px 断点下布局正常
5. `npx tsc --noEmit` 类型检查通过
