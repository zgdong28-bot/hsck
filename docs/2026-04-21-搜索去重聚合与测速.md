## 问题
搜索"我的野蛮奶奶"时：
1. 同名内容来自多个源，全部平铺显示，重复大量
2. 部分源返回不相关结果，混杂其中
3. 无法判断哪个源速度快/质量好，需要逐个点进去试

## 根因分析
1. `searchData()` 直接将每个源的结果追加到 adapter，无去重
2. `matchSearchResult()` 用 `name.contains(searchTitle)` 过滤，精度够但无排序
3. 没有任何源速度/质量评估机制
4. 结果显示顺序取决于哪个源先返回，不是相关度

## 方案

### 核心架构

```
搜索请求 → 5 线程并发 getSearch() → 记录每个源响应时间
                                         ↓
                                    EventBus 回调
                                         ↓
                              searchData() 积累结果
                                         ↓
                              aggregateAndSort():
                                1. 按名称标准化去重
                                2. 相关度打分
                                3. 组内按源速度排序
                                         ↓
                              更新 adapter 显示
```

### 第一步：源测速

在 `searchResult()` 的每个 executor 线程中记录搜索开始时间：

```java
ConcurrentHashMap<String, Long> sourceStartTime = new ConcurrentHashMap<>();
ConcurrentHashMap<String, Long> sourceSpeedMap = new ConcurrentHashMap<>();

// 在 executor 中
sourceStartTime.put(key, System.currentTimeMillis());
sourceViewModel.getSearch(key, searchTitle);
```

在 `searchData()` 中，通过 `video.sourceKey` 计算耗时：

```java
long startTime = sourceStartTime.getOrDefault(sourceKey, 0L);
if (startTime > 0) {
    sourceSpeedMap.put(sourceKey, System.currentTimeMillis() - startTime);
}
```

注意：type 0/1/4 的 getSearch 是异步的（OkGo callback），所以真正的响应时间是从 searchResult() 线程启动到 searchData() EventBus 回调的时间差。这足够作为源速度的代理指标。

### 第二步：结果积累与去重聚合

新增数据结构：

```java
// 聚合后的搜索结果
class AggregatedResult {
    String normalizedName;     // 标准化后的名称
    String displayName;        // 显示用名称
    String type;               // 类型
    String pic;                // 封面图（取第一个有图的）
    int year;                  // 年份
    String note;               // 描述
    float relevanceScore;      // 相关度分数
    List<SourceResult> sources; // 多个源，按速度排序
}

class SourceResult {
    String sourceKey;
    String sourceName;
    long speedMs;              // 搜索响应时间
    Movie.Video video;         // 原始视频对象（用于跳转详情）
}
```

名称标准化逻辑：
- `trim()` 去首尾空白
- 去除全角/半角括号内的标注（如 `(2005)`, `（国语版）`）
- 比较时忽略这些标注

### 第三步：相关度打分

```java
float scoreRelevance(String videoName, String searchTitle) {
    String normalizedName = normalize(videoName);
    String normalizedSearch = normalize(searchTitle);
    
    if (normalizedName.equals(normalizedSearch)) return 1.0f;      // 精确匹配
    if (normalizedName.startsWith(normalizedSearch)) return 0.8f;  // 开头匹配
    if (normalizedName.contains(normalizedSearch)) return 0.6f;    // 包含匹配
    
    // 反向包含（搜索词包含视频名）
    if (normalizedSearch.contains(normalizedName)) return 0.4f;
    
    return 0f; // 不匹配
}
```

### 第四步：聚合排序

```java
void aggregateAndSort() {
    // 1. 按标准化名称分组
    Map<String, AggregatedResult> groups = new LinkedHashMap<>();
    for (Movie.Video video : allResults) {
        String key = normalize(video.name);
        AggregatedResult group = groups.computeIfAbsent(key, k -> new AggregatedResult(...));
        group.sources.add(new SourceResult(video.sourceKey, sourceSpeedMap.get(video.sourceKey), video));
    }
    
    // 2. 组内按速度排序
    for (AggregatedResult group : groups.values()) {
        group.sources.sort(Comparator.comparingLong(s -> s.speedMs));
    }
    
    // 3. 组间按相关度排序
    List<AggregatedResult> sorted = new ArrayList<>(groups.values());
    sorted.sort(Comparator.comparing(r -> -r.relevanceScore));
    
    // 4. 更新 adapter
    aggregatedAdapter.setNewData(sorted);
}
```

### 第五步：UI — 聚合卡片 + 展开选源

**聚合卡片（收起状态）**：
```
┌──────────────────────────────┐
│ [封面图] 我的野蛮奶奶          │
│          喜剧 | 2005 | 共20集  │
│          ▸ 3个源可用 最快120ms  │
└──────────────────────────────┘
```

**展开状态**：
```
┌──────────────────────────────┐
│ [封面图] 我的野蛮奶奶          │
│          喜剧 | 2005 | 共20集  │
│ ┌────────────────────────┐   │
│ │ ⚡ 源A  120ms          │   │
│ │    源B  350ms          │   │
│ │    源C  890ms          │   │
│ └────────────────────────┘   │
└──────────────────────────────┘
```

交互：
- 点击卡片：展开/收起源列表
- 点击某个源：跳转到该源的详情页（复用现有 DetailActivity 跳转逻辑）
- 最快的源前面加 ⚡ 标记

**需要新增的 UI 文件**：
- `item_search_aggregated.xml`：聚合卡片布局
- `item_search_source.xml`：源列表项布局
- `SearchAggregatedAdapter.java`：聚合结果 adapter

### 第六步：渐进式显示

不等所有源搜完再显示——每收到一批结果就重新聚合排序：

```java
private void searchData(AbsXml absXml) {
    // 积累新结果
    accumulateResults(absXml);
    
    // 重新聚合排序并更新 UI
    aggregateAndSort();
    
    // 检查是否全部完成
    if (allRunCount.decrementAndGet() <= 0) {
        // 最终排序（此时所有速度数据都有了）
        aggregateAndSort();
        cancel();
    }
}
```

## 风险点
1. 渐进排序性能：30+ 源时每次重排可能卡顿 — 可用 DiffUtil 优化
2. 名称标准化精度："我的野蛮奶奶" vs "我的野蛮奶奶第1季" — 初版做简单 trim + 精确匹配，后续可加 Jaccard
3. TV 适配：展开/收起交互需要适配 D-pad 焦点导航
4. 源速度准确性：异步回调的时间差是近似值，不是精确的网络延迟。但作为相对排序够用

## 验证方式
- 搜索"我的野蛮奶奶"，确认同名结果聚合为一条
- 确认源按速度排序，最快的有标记
- 确认点击展开后可选择具体源跳转
- 确认不相关结果被排到底部或过滤掉
- 确认 D-pad 焦点导航正常（TV 场景）
- 确认渐进显示流畅，不卡顿
